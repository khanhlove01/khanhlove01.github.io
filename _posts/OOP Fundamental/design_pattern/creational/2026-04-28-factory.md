## The Problem

In software design, you often need to create objects — but the **exact type** of object to create isn't known until runtime, or may vary depending on context.

Imagine an application that needs to draw different shapes, handle different payment methods, or export data in different formats. A naive approach hard-codes the creation logic directly in the calling code:

```python
def get_shape(kind: str):
    if kind == "circle":
        return Circle(radius=5)
    elif kind == "rectangle":
        return Rectangle(width=4, height=6)
    elif kind == "triangle":
        return Triangle(base=3, height=8)
    # Adding a new shape means editing THIS function everywhere it appears
```

This leads to several problems:
1. **Open/Closed Violation:** Every time you add a new type, you must modify existing code, which risks breaking existing behavior.
2. **Tight Coupling:** The caller is tightly coupled to every concrete class it can create.
3. **Duplication:** The same `if/elif` creation logic gets copy-pasted across the codebase.

We need a way to **delegate object creation** to subclasses or specialized components, so the calling code stays clean and stable even as new types are added.

## The Solution: The Factory Method Pattern

The Factory Method pattern defines an **interface for creating an object**, but lets **subclasses decide which class to instantiate**. The creator class provides a template method that calls the factory method — subclasses override just that one method to swap in a different product.

The key participants are:
1. **Product (Abstract):** The interface all concrete products implement.
2. **Concrete Products:** The actual objects being created (e.g., `Circle`, `CreditCardPayment`).
3. **Creator (Abstract):** Declares the factory method and may provide a template method that uses it.
4. **Concrete Creators:** Override the factory method to return a specific product.

> **Why not just use `if/elif`?** The factory method moves the branching into the class hierarchy. Each concrete creator encapsulates one choice, making it trivially easy to add a new product type by adding a new creator — without touching any existing code.

Here are four real-world scenarios demonstrating the Factory Method pattern in Python.

### Example 1: Shape Creator

The classic introduction. A `ShapeCreator` defines a template method `describe()` that calls `create_shape()` — a factory method overridden by each subclass to return the right shape.

```python
import math
from abc import ABC, abstractmethod


class Shape(ABC):
    @abstractmethod
    def area(self) -> float:
        pass

    @abstractmethod
    def describe(self) -> None:
        pass


class Circle(Shape):
    def __init__(self, radius: float) -> None:
        self.radius: float = radius

    def area(self) -> float:
        return math.pi * self.radius * self.radius

    def describe(self) -> None:
        print(f"Circle with area: {self.area():.2f}")


class Rectangle(Shape):
    def __init__(self, width: float, height: float) -> None:
        self.width: float = width
        self.height: float = height

    def area(self) -> float:
        return self.width * self.height

    def describe(self) -> None:
        print(f"Rectangle with area: {self.area():.2f}")


class Triangle(Shape):
    def __init__(self, base: float, height: float) -> None:
        self.base: float = base
        self.height: float = height

    def area(self) -> float:
        return 0.5 * self.base * self.height

    def describe(self) -> None:
        area = round(self.area(), 2)
        print(f"Triangle with area: {area}")


class ShapeCreator(ABC):
    @abstractmethod
    def create_shape(self) -> Shape:
        pass

    def describe(self) -> None:
        shape = self.create_shape()
        shape.describe()


class CircleCreator(ShapeCreator):
    def create_shape(self) -> Shape:
        return Circle(radius=5)


class RectangleCreator(ShapeCreator):
    def create_shape(self) -> Shape:
        return Rectangle(width=4, height=6)


class TriangleCreator(ShapeCreator):
    def create_shape(self) -> Shape:
        return Triangle(base=3, height=8)


if __name__ == "__main__":
    creator: ShapeCreator = CircleCreator()
    creator.describe()

    creator = RectangleCreator()
    creator.describe()

    creator = TriangleCreator()
    creator.describe()
```

Each `Creator` subclass overrides only `create_shape()`. The `describe()` template method in the base class never needs to change, even if you add a `HexagonCreator` tomorrow.

### Example 2: Payment Processor

A checkout flow that needs to support multiple payment methods. Each `PaymentCreator` encapsulates one payment strategy. The `checkout()` template method handles the full flow (validate → process → receipt) without knowing which payment type it's working with.

```python
from abc import ABC, abstractmethod


class PaymentMethod(ABC):
    @abstractmethod
    def validate_payment(self) -> bool:
        pass

    @abstractmethod
    def process_payment(self, amount: float) -> None:
        pass

    @abstractmethod
    def get_receipt(self) -> str:
        pass


class CreditCardPayment(PaymentMethod):
    def validate_payment(self) -> bool:
        print("Validating credit card...")
        return True

    def process_payment(self, amount: float) -> None:
        fee: float = amount * 0.025
        print(f"Processing credit card payment: $[{amount:.2f}] (fee: ${fee:.2f})")

    def get_receipt(self) -> str:
        return "Card ending in ****1234"


class PayPalPayment(PaymentMethod):
    def validate_payment(self) -> bool:
        print("Validating PayPal account...")
        return True

    def process_payment(self, amount: float) -> None:
        fee: float = amount * 0.015
        print(f"Processing PayPal payment: $[{amount:.2f}] (fee: ${fee:.2f})")

    def get_receipt(self) -> str:
        return "PayPal: user@email.com"


class CryptoPayment(PaymentMethod):
    def validate_payment(self) -> bool:
        print("Validating crypto wallet...")
        return True

    def process_payment(self, amount: float) -> None:
        fee: float = amount * 0.005
        print(f"Processing crypto payment: $[{amount:.2f}] (fee: ${fee:.2f})")

    def get_receipt(self) -> str:
        return "Wallet: 0x1a2b...3c4d"


class PaymentCreator(ABC):
    @abstractmethod
    def create_payment(self) -> PaymentMethod:
        pass

    def checkout(self, amount: float) -> None:
        payment = self.create_payment()
        if not payment.validate_payment():
            print("Payment validation failed.")
            return
        payment.process_payment(amount=amount)
        print(f"Receipt: {payment.get_receipt()}")


class CreditCardPaymentCreator(PaymentCreator):
    def create_payment(self) -> PaymentMethod:
        return CreditCardPayment()


class PayPalPaymentCreator(PaymentCreator):
    def create_payment(self) -> PaymentMethod:
        return PayPalPayment()


class CryptoPaymentCreator(PaymentCreator):
    def create_payment(self) -> PaymentMethod:
        return CryptoPayment()


if __name__ == "__main__":
    processor = CreditCardPaymentCreator()
    processor.checkout(100.00)

    processor = PayPalPaymentCreator()
    processor.checkout(250.00)

    processor = CryptoPaymentCreator()
    processor.checkout(500.00)
```

Notice how `checkout()` is completely agnostic to the payment type. Adding a new `BankTransferPayment` only requires a new product class and a new creator — zero changes to existing code.

### Example 3: Plugin System with a Registry

A more advanced pattern: combining Factory Method with a **runtime registry**. Instead of a long `if/elif` chain, creators are registered by name and looked up dynamically. This is ideal for plugin architectures where implementations are selected from config files or feature flags.

```python
from abc import ABC, abstractmethod
from typing import Dict


class Plugin(ABC):
    @abstractmethod
    def initialize(self) -> None:
        pass

    @abstractmethod
    def execute(self, input_str: str) -> str:
        pass

    @abstractmethod
    def get_name(self) -> str:
        pass


class LoggerPlugin(Plugin):
    def initialize(self) -> None:
        print("[Logger] Initialized.")

    def execute(self, input_str: str) -> str:
        print(f"[Logger] Logging: {input_str}")
        return input_str

    def get_name(self) -> str:
        return "Logger"


class ValidatorPlugin(Plugin):
    def initialize(self) -> None:
        print("[Validator] Initialized.")

    def execute(self, input_str: str) -> str:
        ok: bool = len(input_str) > 0
        status: str = "valid" if ok else "invalid"
        print(f'[Validator] Input "{input_str}" is {status}.')
        return "valid" if ok else "invalid"

    def get_name(self) -> str:
        return "Validator"


class TransformerPlugin(Plugin):
    def initialize(self) -> None:
        print("[Transformer] Initialized.")

    def execute(self, input_str: str) -> str:
        upper: str = input_str.upper()
        print(f"[Transformer] Result: {upper}")
        return upper

    def get_name(self) -> str:
        return "Transformer"


class PluginCreator(ABC):
    @abstractmethod
    def create_plugin(self) -> Plugin:
        pass

    def run(self, input_str: str) -> None:
        plugin = self.create_plugin()
        plugin.initialize()
        plugin.execute(input_str)
        print(f'Plugin "{plugin.get_name()}" executed.')
        print()


class LoggerPluginCreator(PluginCreator):
    def create_plugin(self) -> Plugin:
        return LoggerPlugin()


class ValidatorPluginCreator(PluginCreator):
    def create_plugin(self) -> Plugin:
        return ValidatorPlugin()


class TransformerPluginCreator(PluginCreator):
    def create_plugin(self) -> Plugin:
        return TransformerPlugin()


class PluginRegistry:
    """Maps names to concrete PluginCreator instances.

    Factory Method hides which Plugin subclass each creator builds. A registry
    lets you choose that creator at runtime from config, CLI, or feature flags
    instead of a long if/elif chain over kinds.
    """

    def __init__(self) -> None:
        self._creators: Dict[str, PluginCreator] = {}

    def register(self, name: str, creator: PluginCreator) -> None:
        self._creators[name] = creator

    def get_creator(self, name: str) -> PluginCreator:
        if name not in self._creators:
            raise KeyError(f"Plugin not registered: {name}")
        return self._creators[name]

    def run(self, name: str, input_str: str) -> None:
        self.get_creator(name).run(input_str)


if __name__ == "__main__":
    registry: PluginRegistry = PluginRegistry()
    registry.register(name="logger", creator=LoggerPluginCreator())
    registry.register(name="validator", creator=ValidatorPluginCreator())
    registry.register(name="transformer", creator=TransformerPluginCreator())

    for name in ["logger", "validator", "transformer"]:
        registry.run(name, "Hello World")

    # Same as choosing implementation at runtime, e.g. user_config["plugin"]:
    chosen = "transformer"
    registry.run(chosen, "runtime pick")
```

The `PluginRegistry` owns a `Dict[str, PluginCreator]`. The caller never imports a concrete plugin class — it only knows the string key. To add a new plugin, you register a new creator. Nothing else changes.

### Example 4: Data Exporter with a Hardcoded Registry

A simpler variant of the registry idea: a **hardcoded dict** mapping format names to creator instances. This is a clean pattern when the set of valid keys is fixed and known at compile time (e.g., `"csv"` vs `"json"`).

```python
from __future__ import annotations

import json
from abc import ABC, abstractmethod
from typing import Dict, Final, List


class ExportFormatter(ABC):
    @abstractmethod
    def format_rows(self, headers: List[str], rows: List[List[str]]) -> str:
        pass


class CsvExportFormatter(ExportFormatter):
    def format_rows(self, headers: List[str], rows: List[List[str]]) -> str:
        lines: List[str] = [",".join(headers)]
        for row in rows:
            cells: List[str] = row
            lines.append(",".join(cells))
        return "\n".join(lines)


class JsonExportFormatter(ExportFormatter):
    def format_rows(self, headers: List[str], rows: List[List[str]]) -> str:
        objects: List[Dict[str, str]] = [
            dict(zip(headers, row)) for row in rows
        ]
        return json.dumps(obj=objects, indent=2)


class ExportJobCreator(ABC):
    @abstractmethod
    def create_formatter(self) -> ExportFormatter:
        pass

    def run_export(self, headers: List[str], rows: List[List[str]]) -> None:
        formatter: ExportFormatter = self.create_formatter()
        text: str = formatter.format_rows(headers=headers, rows=rows)
        print(text)
        print(f"(exported via {self.__class__.__name__})")


class CsvExportJobCreator(ExportJobCreator):
    def create_formatter(self) -> ExportFormatter:
        return CsvExportFormatter()


class JsonExportJobCreator(ExportJobCreator):
    def create_formatter(self) -> ExportFormatter:
        return JsonExportFormatter()


# Hardcoded registry: format name -> creator instance.
EXPORT_JOB_CREATORS: Final[Dict[str, ExportJobCreator]] = {
    "csv": CsvExportJobCreator(),
    "json": JsonExportJobCreator(),
}


def export_data(kind: str, headers: List[str], rows: List[List[str]]) -> None:
    """Pick a factory from the hardcoded dict, then run the template method."""
    if kind not in EXPORT_JOB_CREATORS:
        raise KeyError(f"Unknown export kind: {kind!r}; known: {list(EXPORT_JOB_CREATORS)}")
    EXPORT_JOB_CREATORS[kind].run_export(headers=headers, rows=rows)


if __name__ == "__main__":
    headers: List[str] = ["sku", "qty", "price"]
    rows: List[List[str]] = [
        ["A1", "2", "9.99"],
        ["B2", "1", "19.50"],
    ]

    print("=== CSV (from registry key 'csv') ===")
    export_data(kind="csv", headers=headers, rows=rows)

    print("\n=== JSON (from registry key 'json') ===")
    export_data(kind="json", headers=headers, rows=rows)

    print("\n=== Same pattern: choose key as if from config ===")
    user_choice: str = "json"
    export_data(kind=user_choice, headers=headers, rows=rows)
```

The `EXPORT_JOB_CREATORS` dict acts as an immutable registry. The `export_data()` function is a thin dispatcher — it finds the right creator from the dict and calls the `run_export()` template method. Adding XML export means adding one class and one dict entry.

## Simple Factory vs Factory Method

You might be wondering: *"Where is the Factory class?"* — as in, a dedicated class with a `create()` method that takes a string and returns the right object. That's actually a **different pattern** called **Simple Factory**, and it's worth understanding the distinction.

### What is Simple Factory?

Simple Factory is **not a GoF design pattern** — it's a common programming idiom. It centralizes object creation into a single class with a static (or class) method:

```python
class ShapeFactory:
    @staticmethod
    def create(kind: str) -> Shape:
        if kind == "circle":
            return Circle(radius=5)
        elif kind == "rectangle":
            return Rectangle(width=4, height=6)
        elif kind == "triangle":
            return Triangle(base=3, height=8)
        raise ValueError(f"Unknown shape: {kind}")

# Usage:
shape = ShapeFactory.create("circle")
shape.describe()
```

### Key Differences

| | Simple Factory | Factory Method |
|---|---|---|
| **Factory location** | One dedicated class | Each Creator subclass *is* the factory |
| **Adding a new type** | Must edit the Factory class | Add a new subclass only |
| **Open/Closed Principle** | ❌ Violated | ✅ Respected |
| **Is a GoF pattern?** | ❌ No | ✅ Yes |
| **When to use** | Fixed, small set of types | Open-ended, extensible hierarchies |

> **Rule of thumb:** If you know the set of types will rarely change (e.g., 2–3 fixed formats), Simple Factory is perfectly fine and easier to understand. If the set of types is expected to grow, use Factory Method.

## Simple Factory Examples

The same four domains, rewritten using a dedicated Factory class instead of creator subclasses.

### Example 8: Export Factory (Simple Factory)

A `FormatterFactory` centralizes all format-string-to-formatter mappings. Clean and simple — ideal when the supported formats are a fixed closed set.

            lines.append(",".join(row))

## Conclusion

Both patterns solve the same surface problem — "how do I create the right object?" — but from different angles:

- **Simple Factory** is pragmatic. One class, one `if/elif` block, easy to read. Perfect for a fixed, small set of types that rarely change.
- **Factory Method** is extensible. Creation logic is distributed across a class hierarchy. Adding a new type never touches existing code.

| Scenario | Recommended Pattern |
|---|---|
| Fixed 2–3 known types, unlikely to grow | Simple Factory |
| Open-ended set, new types added frequently | Factory Method |
| Runtime-configurable (config, plugins, CLI) | Factory Method + Registry |

The pattern shines whenever you find yourself writing `if kind == "A": return A()` blocks that keep growing — that's the signal to move from Simple Factory to Factory Method.
