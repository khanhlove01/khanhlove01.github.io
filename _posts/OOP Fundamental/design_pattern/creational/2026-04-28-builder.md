## The Problem

Some objects are complex — they have many fields, some required, some optional, some with defaults, and some that only make sense in combination with others. Constructing them with a regular constructor quickly becomes painful:

```python
# What does True, False, "high", None even mean here?
email = Email("tmk@gmail.com", "Meeting", [], [], "high", True, False, None)
```

This is the **telescoping constructor anti-pattern**: as you add more fields, the constructor grows, callers must remember the order of every argument, and optional fields force awkward `None` placeholders.

You might try keyword arguments to help readability:

```python
email = Email(
    to="tmk@gmail.com",
    subject="Meeting",
    priority="high",
    cc=None,
    bcc=None,
    body=None,
)
```

Better — but still problematic:
1. **Validation is hard:** When is the object checked for required fields? You can't easily enforce rules like "a report must have at least one section" at construction time without cluttering `__init__`.
2. **Object is mutable during setup:** The object exists in a half-built state while you set fields one by one, which can cause bugs if it's accidentally used before it's ready.
3. **No fluent chaining:** Setting fields requires multiple statements; there's no natural reading flow.

We need a way to **construct complex objects step by step**, keeping the object immutable until it is fully ready, with a readable, chainable API.

## The Solution: The Builder Pattern

The Builder pattern separates the **construction** of an object from its **representation**. A dedicated `Builder` inner class collects all the configuration via fluent setter methods, then produces the fully initialized, immutable product in a single `build()` call.

The key participants are:
1. **Product:** The complex object being built (e.g., `Email`, `Pizza`, `Report`). Its constructor takes a builder — never raw parameters.
2. **Builder:** Holds the mutable state during construction. Each setter returns `self` to enable method chaining. `build()` validates and returns the finished product.
3. **Director (optional):** A higher-level class that knows *how* to drive the builder for common presets. Useful when the same configurations are reused across the codebase.

> **Why an inner class?** Placing `Builder` inside the product class (e.g., `Email.Builder`) keeps them co-located and makes the API self-documenting: `Email.Builder(...)...build()` reads like a sentence.

Here are three real-world scenarios demonstrating the Builder pattern in Python.

### Example 1: Email Builder

Building an email message with many optional fields — CC, BCC, body, priority, and attachments. Required fields (`to`, `subject`) are constructor parameters of the builder; everything else is optional and has sensible defaults.

{% raw %}
```python
from typing import List, Optional, Self


class Email:
    def __init__(self, builder: 'Builder') -> None:
        self.to: str = builder._to
        self.subject: str = builder._subject
        self.cc: List[str] = list(builder._cc)
        self.bcc: List[str] = list(builder._bcc)
        self.body: Optional[str] = builder._body
        self.priority: str = builder._priority
        self.attachments: List[str] = list(builder._attachments)

    def __str__(self) -> str:
        def _sq(s: str) -> str:
            return "'" + s.replace("\\", "\\\\").replace("'", "\\'") + "'"

        def _lst(items: List[str]) -> str:
            return "[" + ", ".join(items) + "]"

        if self.body is None:
            body_part = "body=None"
        else:
            body_part = f"body={_sq(self.body)}"

        return (
            f"Email{{to={_sq(self.to)}, subject={_sq(self.subject)}, cc={_lst(self.cc)}, "
            f"bcc={_lst(self.bcc)}, {body_part}, priority={_sq(self.priority)}, "
            f"attachments={_lst(self.attachments)}}}"
        )

    class Builder:
        def __init__(self, to: str, subject: str) -> None:
            self._to: str = to
            self._subject: str = subject
            self._cc: List[str] = []
            self._bcc: List[str] = []
            self._body: Optional[str] = None
            self._priority: str = "normal"
            self._attachments: List[str] = []

        def cc(self, cc: str) -> Self:
            self._cc.append(cc)
            return self

        def bcc(self, bcc: str) -> Self:
            self._bcc.append(bcc)
            return self

        def body(self, body: str) -> Self:
            self._body = body
            return self

        def priority(self, priority: str) -> Self:
            self._priority = priority
            return self

        def attachment(self, attachment: str) -> Self:
            self._attachments.append(attachment)
            return self

        def build(self) -> 'Email':
            return Email(builder=self)


if __name__ == "__main__":
    email1: Email = Email.Builder(
        to="tmk@gmail.com",
        subject="Meeting Tomorrow",
    ).body(body="Meeting at 10AM in conference room B")\
    .priority(priority="Highest").build()

    email2: Email = (
        Email.Builder(to="team@company.com", subject="Q1 deck review")
        .cc(cc="lead@company.com")
        .cc(cc="pm@company.com")
        .bcc(bcc="archive@company.com")
        .bcc(bcc="audit@company.com")
        .attachment(attachment="slides_q1.pdf")
        .attachment(attachment="notes.docx")
        .body(body="Please review the attached deck before Friday.")
        .priority(priority="high")
        .build()
    )

    print(email1)
    print(email2)
```
{% endraw %}

Notice how `email1` and `email2` use completely different combinations of optional fields — yet the `Email` constructor itself never changes. The builder absorbs all the variability.

### Example 2: Pizza Builder with a Director

A pizza order has a fixed structure (size, crust, sauce, cheese, toppings) but enormous combinatorial variety. Here we add a `PizzaDirector` that encodes common recipes — so callers can get a `Margherita` in one line, or bypass the director entirely to build a fully custom pizza.

{% raw %}
```python
from __future__ import annotations

from typing import List, Self


class Pizza:
    def __init__(self, builder: Builder) -> None:
        self.size: str = builder._size
        self.crust: str = builder._crust
        self.sauce: str = builder._sauce
        self.cheese: str = builder._cheese
        self.toppings: List[str] = list(builder._toppings)

    def __str__(self) -> str:
        def _sq(s: str) -> str:
            return "'" + s.replace("\\", "\\\\").replace("'", "\\'") + "'"

        toppings_fmt = "[" + ", ".join(self.toppings) + "]"
        return (
            f"Pizza{{size={_sq(self.size)}, crust={_sq(self.crust)}, "
            f"sauce={_sq(self.sauce)}, cheese={_sq(self.cheese)}, "
            f"toppings={toppings_fmt}}}"
        )

    class Builder:
        def __init__(self, size: str) -> None:
            self._size: str = size
            self._crust: str = "regular"
            self._sauce: str = "tomato"
            self._cheese: str = "mozzarella"
            self._toppings: List[str] = []

        def crust(self, crust: str) -> Self:
            self._crust = crust
            return self

        def sauce(self, sauce: str) -> Self:
            self._sauce = sauce
            return self

        def cheese(self, cheese: str) -> Self:
            self._cheese = cheese
            return self

        def add_topping(self, topping: str) -> Self:
            self._toppings.append(topping)
            return self

        def build(self) -> Pizza:
            return Pizza(builder=self)


class PizzaDirector:
    def build_margherita(self, size: str) -> Pizza:
        return (
            Pizza.Builder(size=size)
            .crust(crust="regular")
            .sauce(sauce="tomato")
            .cheese(cheese="mozzarella")
            .add_topping(topping="basil")
            .build()
        )

    def build_pepperoni(self, size: str) -> Pizza:
        return (
            Pizza.Builder(size=size)
            .crust(crust="thin")
            .sauce(sauce="tomato")
            .cheese(cheese="mozzarella")
            .add_topping(topping="pepperoni")
            .add_topping(topping="olives")
            .build()
        )

    def build_veggie(self, size: str) -> Pizza:
        return (
            Pizza.Builder(size=size)
            .crust(crust="whole wheat")
            .sauce(sauce="pesto")
            .cheese(cheese="gouda")
            .add_topping(topping="mushrooms")
            .add_topping(topping="peppers")
            .add_topping(topping="onions")
            .add_topping(topping="olives")
            .build()
        )


if __name__ == "__main__":
    director: PizzaDirector = PizzaDirector()
    margherita: Pizza = director.build_margherita(size="medium")
    pepperoni: Pizza = director.build_pepperoni(size="large")
    veggie: Pizza = director.build_veggie(size="small")

    custom: Pizza = (
        Pizza.Builder(size="large")
        .crust(crust="stuffed")
        .sauce(sauce="bbq")
        .cheese(cheese="cheddar")
        .add_topping(topping="chicken")
        .add_topping(topping="bacon")
        .add_topping(topping="jalapenos")
        .build()
    )

    print(margherita)
    print(pepperoni)
    print(veggie)
    print(custom)
```
{% endraw %}

The `PizzaDirector` is entirely optional. Callers who need a standard recipe delegate to it; callers who need a one-off configuration go straight to `Pizza.Builder`. The builder doesn't know or care which path was taken.

### Example 3: Report Builder with Validation

A structured report with a title, optional subtitle, ordered sections, optional charts, and an optional footer. This example highlights **build-time validation**: the builder enforces that a report must have at least one section before `build()` returns — something impossible to enforce cleanly with a plain constructor.

```python
from __future__ import annotations

from typing import List, Optional, Self, Tuple


class Report:
    _LINE_WIDTH = 72

    def __init__(self, builder: Report.Builder) -> None:
        self.title: str = builder._title
        self.subtitle: Optional[str] = builder._subtitle
        self.sections: List[Tuple[str, str]] = list(builder._sections)
        self.charts: List[str] = list(builder._charts)
        self.footer: Optional[str] = builder._footer

    def __str__(self) -> str:
        w = Report._LINE_WIDTH
        parts: List[str] = []

        parts.append("=" * w)
        parts.append(self.title)
        if self.subtitle:
            parts.append(self.subtitle)
        parts.append("=" * w)
        parts.append("")

        for i, (heading, content) in enumerate(self.sections, start=1):
            parts.append(f"{i}. {heading}")
            for line in content.splitlines() or [""]:
                parts.append(f"    {line}")
            parts.append("")

        if self.charts:
            parts.append("Charts:")
            for chart in self.charts:
                parts.append(f"  - {chart}")
            parts.append("")

        if self.footer:
            parts.append(f"--- {self.footer} ---")

        return "\n".join(parts).rstrip() + "\n"

    class Builder:
        def __init__(self, title: str) -> None:
            self._title: str = title
            self._subtitle: Optional[str] = None
            self._sections: List[Tuple[str, str]] = []
            self._charts: List[str] = []
            self._footer: Optional[str] = None

        def subtitle(self, subtitle: str) -> Self:
            self._subtitle = subtitle
            return self

        def add_section(self, heading: str, content: str) -> Self:
            self._sections.append((heading, content))
            return self

        def add_chart(self, chart: str) -> Self:
            self._charts.append(chart)
            return self

        def footer(self, footer: str) -> Self:
            self._footer = footer
            return self

        def build(self) -> Report:
            if not self._sections:
                raise ValueError("Report must have at least one section")
            return Report(builder=self)


if __name__ == "__main__":
    report: Report = (
        Report.Builder(title="Q4 Performance Report")
        .subtitle(subtitle="October - December 2024")
        .add_section(heading="Executive Summary", content="Revenue grew 15% quarter over quarter.")
        .add_section(heading="Key Metrics", content="DAU: 1.2M, MAU: 5.8M, Revenue: $12.3M")
        .add_section(heading="Challenges", content="Infrastructure costs increased by 8%.")
        .add_chart(chart="revenue-trend")
        .add_chart(chart="user-growth")
        .footer(footer="Confidential - Internal Use Only")
        .build()
    )

    print(report)

    try:
        Report.Builder(title="Empty Report").build()
    except ValueError as e:
        print(f"Error: {e}")
```

The `build()` method acts as a guard: it raises `ValueError` if the report would be invalid. This means an `Report` object, once created, is **always guaranteed to be well-formed** — callers never need to check for missing sections after the fact.

## Builder vs. Director: When to Use Each

| | Builder alone | Builder + Director |
|---|---|---|
| **Use when** | Each call needs unique configuration | Common preset configurations exist |
| **Flexibility** | Maximum — caller controls every step | Fixed presets + escape hatch for custom |
| **Example** | `email1` and `email2` above | `PizzaDirector.build_margherita()` |

The Director is purely optional. Think of it as a **named constructor** that encodes institutional knowledge ("this is how we build a Margherita") on top of the flexible builder API.

## Conclusion

The Builder pattern shines whenever you have:
- **Many optional fields** that would bloat a constructor with `None` placeholders
- **Validation rules** that span multiple fields and should be checked at build time
- **Repeated configurations** that benefit from a Director encoding named presets

It produces readable, chainable code (`Builder(...).field1().field2().build()`) and guarantees that the resulting object is fully initialized and valid — never a half-built shell.
