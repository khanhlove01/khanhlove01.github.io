## The Problem

Something interesting happens in your system — a stock price changes, a file is modified, a sensor publishes a new reading — and *several other parts* need to react. The naive approach is for the source to call each consumer directly. That works for two consumers, then becomes a tangle.

A concrete example: a stock exchange that updates a price display, an alert service, and a trading bot. The first attempt:

```python
class StockExchange:
    def __init__(self) -> None:
        self.display = PriceDisplay()
        self.alerts  = AlertService()
        self.bot     = TradingBot()

    def update_price(self, symbol: str, price: float) -> None:
        self._prices[symbol] = price
        self.display.refresh(symbol, price)
        self.alerts.check(symbol, price)
        self.bot.consider(symbol, price)
```

This compiles. It even works. The serious problems:

1. **The exchange knows about every consumer:** Adding a portfolio tracker means editing `StockExchange`. The class violates Open/Closed.
2. **The exchange's job has bloated:** It used to track prices; now it also drives a display, an alerter, and a bot.
3. **Consumers can't be added or removed at runtime:** The set is hardcoded. There's no way to add an observer for one session only and detach it after.

## The Solution: The Observer Pattern

The Observer pattern flips the dependency. The source — called the **Subject** — keeps a list of registered **Observers**. When something changes, the subject calls a single method on each observer. The subject knows nothing about its observers beyond the interface.

The key participants are:

1. **Subject:** Holds the data and a list of registered observers. Exposes `register_observer` / `remove_observer` and a private `_notify_observers` that loops over the list.
2. **Observer (Interface):** Declares the callback method (e.g., `on_price_update`, `update`).
3. **Concrete Observers:** Implement the callback. Each one decides what to do with the notification independently.
4. **Client:** Constructs the subject and observers, registers the observers, and triggers state changes on the subject.

> **Push vs Pull:** The subject can either *push* the data into the callback (`on_update(price)`) or *pull* it via a back-reference (`on_update(subject)` and the observer queries what it needs). Pull is more flexible — different observers care about different fields — and is what we use below.

Three real-world scenarios in Python.

### Example 1: Stock Exchange (Pull-Style Notification)

The exchange holds the price map and notifies registered observers when something changes. Each observer queries the exchange for what it cares about — the current price, the last symbol updated. Adding a new observer is a single new class.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class StockObserver(ABC):
    @abstractmethod
    def on_price_update(self, exchange: StockExchange) -> None: ...


class StockExchange:
    def __init__(self) -> None:
        self._prices: dict[str, float] = {}
        self._observers: list[StockObserver] = []
        self._last_symbol: str | None = None

    def register(self, observer: StockObserver) -> None:
        self._observers.append(observer)

    def unregister(self, observer: StockObserver) -> None:
        if observer in self._observers:
            self._observers.remove(observer)

    def update_price(self, symbol: str, price: float) -> None:
        self._prices[symbol] = price
        self._last_symbol = symbol
        for o in list(self._observers):
            o.on_price_update(self)

    def get_price(self, symbol: str) -> float:
        return self._prices.get(symbol, 0.0)

    @property
    def last_symbol(self) -> str | None:
        return self._last_symbol


class PriceDisplay(StockObserver):
    def on_price_update(self, exchange: StockExchange) -> None:
        symbol = exchange.last_symbol
        print(f"Display -> {symbol}: ${exchange.get_price(symbol):.2f}")


class AlertService(StockObserver):
    def __init__(self) -> None:
        self._thresholds: dict[str, float] = {}

    def set_alert(self, symbol: str, threshold: float) -> None:
        self._thresholds[symbol] = threshold

    def on_price_update(self, exchange: StockExchange) -> None:
        symbol = exchange.last_symbol
        if symbol in self._thresholds:
            price = exchange.get_price(symbol)
            if price >= self._thresholds[symbol]:
                print(f"ALERT -> {symbol} hit ${price:.2f}")


class TradingBot(StockObserver):
    def __init__(self) -> None:
        self._previous: dict[str, float] = {}

    def on_price_update(self, exchange: StockExchange) -> None:
        symbol = exchange.last_symbol
        current = exchange.get_price(symbol)
        previous = self._previous.get(symbol, current)
        if current < previous:
            print(f"Bot -> {symbol} dropped (${previous:.2f} -> ${current:.2f}). BUY.")
        self._previous[symbol] = current


if __name__ == "__main__":
    exchange = StockExchange()
    exchange.register(PriceDisplay())

    alerts = AlertService()
    alerts.set_alert("AAPL", 180.0)
    exchange.register(alerts)

    exchange.register(TradingBot())

    exchange.update_price("AAPL", 175.50)
    exchange.update_price("AAPL", 182.00)
```

Iterating over `list(self._observers)` (a *copy*) defends against the case where an observer's callback mutates the registration list — a small but classic re-entrance trap.

### Example 2: Weather Station (Classic GoF Push-Style)

The weather station notifies observers of new measurements; each observer interprets them differently. `CurrentConditionsDisplay` shows the latest reading; `StatisticsDisplay` accumulates a rolling average. Same notification, different state.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class WeatherObserver(ABC):
    @abstractmethod
    def update(self, station: WeatherStation) -> None: ...


class WeatherStation:
    def __init__(self) -> None:
        self.temperature = 0.0
        self.humidity = 0.0
        self.pressure = 0.0
        self._observers: list[WeatherObserver] = []

    def register(self, observer: WeatherObserver) -> None:
        self._observers.append(observer)

    def set_measurements(self, temp: float, humidity: float, pressure: float) -> None:
        self.temperature, self.humidity, self.pressure = temp, humidity, pressure
        for o in list(self._observers):
            o.update(self)


class CurrentConditionsDisplay(WeatherObserver):
    def update(self, s: WeatherStation) -> None:
        print(f"Now -> Temp: {s.temperature}°C, Humidity: {s.humidity}%, "
              f"Pressure: {s.pressure} hPa")


class StatisticsDisplay(WeatherObserver):
    def __init__(self) -> None:
        self._readings: list[float] = []

    def update(self, s: WeatherStation) -> None:
        self._readings.append(s.temperature)
        avg = sum(self._readings) / len(self._readings)
        print(f"Stats -> Avg temperature: {avg:.1f}°C")


if __name__ == "__main__":
    station = WeatherStation()
    station.register(CurrentConditionsDisplay())
    station.register(StatisticsDisplay())

    station.set_measurements(25.0, 65.0, 1013.0)
    station.set_measurements(28.0, 70.0, 1012.0)
    station.set_measurements(22.0, 90.0, 1011.0)
```

Each observer keeps its own state — the statistics display accumulates readings; the current-conditions display does not. Observers are **independent** of each other; the subject knows nothing about their internals.

### Example 3: File-System Monitoring (Typed Events)

A more realistic shape: instead of pushing the entire subject, the source emits a *typed event* describing what happened. Observers filter by event type. This scales when you have many event kinds and observers that care about subsets.

```python
from __future__ import annotations
from abc import ABC, abstractmethod
from datetime import datetime
from enum import Enum


class FileEventType(Enum):
    CREATED  = "CREATED"
    MODIFIED = "MODIFIED"
    DELETED  = "DELETED"


class FileEvent:
    def __init__(self, event_type: FileEventType, filename: str) -> None:
        self.type = event_type
        self.filename = filename
        self.timestamp = datetime.now()


class FileEventListener(ABC):
    @abstractmethod
    def on_file_event(self, event: FileEvent) -> None: ...


class FileWatcher:
    def __init__(self) -> None:
        self._listeners: list[FileEventListener] = []

    def add_listener(self, listener: FileEventListener) -> None:
        self._listeners.append(listener)

    def emit(self, event_type: FileEventType, filename: str) -> None:
        event = FileEvent(event_type, filename)
        print(f"\n[{event.timestamp:%H:%M:%S}] {event_type.value}: {filename!r}")
        for listener in list(self._listeners):
            listener.on_file_event(event)


class BackupService(FileEventListener):
    def on_file_event(self, event: FileEvent) -> None:
        if event.type in (FileEventType.CREATED, FileEventType.MODIFIED):
            print(f"  Backup    -> {event.filename!r}")


class AuditLogger(FileEventListener):
    def on_file_event(self, event: FileEvent) -> None:
        print(f"  Audit     -> [{event.type.value}] {event.filename!r}")


class CleanupService(FileEventListener):
    def on_file_event(self, event: FileEvent) -> None:
        if event.type is FileEventType.DELETED:
            print(f"  Cleanup   -> Removing artefacts of {event.filename!r}")


if __name__ == "__main__":
    watcher = FileWatcher()
    watcher.add_listener(BackupService())
    watcher.add_listener(AuditLogger())
    watcher.add_listener(CleanupService())

    watcher.emit(FileEventType.CREATED,  "report.pdf")
    watcher.emit(FileEventType.MODIFIED, "report.pdf")
    watcher.emit(FileEventType.DELETED,  "old-data.csv")
```

`AuditLogger` cares about every event; the others filter on type. The filter logic lives where it belongs — *inside the observer* — not in the subject.

## When to Use the Observer Pattern

| Situation | Without Observer | With Observer |
|---|---|---|
| One change → many reactions | Subject calls each consumer directly | Subject loops over registered observers |
| Set of consumers changes at runtime | Hardcoded — can't add/remove | `register` / `unregister` on the fly |
| Subject and consumers in different layers | Direct dependency, awkward to break | Consumers depend on subject; subject only knows the interface |
| Adding a new reaction | Modify the subject | Add one new observer class |

## Conclusion

The Observer pattern is the right tool when:

- **A change has multiple, independent downstream effects:** And the subject shouldn't be coupled to all of them.
- **The set of listeners is dynamic:** They register, unregister, sometimes only for one session.
- **The listeners are independent:** They don't talk to each other; they each just react to the notification.

The pattern's strength is **decoupling**. The subject's job stays small — track its own state and notify. The observers' jobs stay focused — react to one event. Adding the third, fourth, fifth listener costs nothing in the subject.

Caveats to keep in mind:

- **Re-entrance:** If an observer's callback ends up calling back into the subject (e.g., placing a counter-bid that triggers a new notification), iterate over a *copy* of the observer list to avoid mutation-during-iteration. Some implementations also add a `_notifying` flag to short-circuit recursive notifications.
- **Memory:** A subject holds strong references to its observers. If observers are short-lived, make sure they `unregister` themselves — or use weak references — so the subject doesn't keep them alive.
- **Order:** The pattern doesn't guarantee delivery order. If two observers must run in a specific order, you have an implicit dependency that the pattern is hiding from you — refactor.
