## The Problem

You have an object — a button, a remote control, a menu item — that triggers actions. The naive approach is to hardcode the action: `button.on_press()` calls `light.on()` directly. That works for one action; the moment you need *undo*, *redo*, *queueing*, *logging*, or *replay*, the model collapses.

A concrete example: a smart-home remote with several actions. The first attempt:

```python
class RemoteControl:
    def __init__(self, light: Light, thermostat: Thermostat) -> None:
        self.light = light
        self.thermostat = thermostat

    def press_a(self) -> None: self.light.on()
    def press_b(self) -> None: self.light.off()
    def press_c(self) -> None: self.thermostat.set_temperature(25)
```

This works. Now make these buttons configurable per user. Now add an undo button. The serious problems:

1. **Actions are not data:** They're hardcoded calls. You can't put them in a list, queue them, schedule them, or replay them.
2. **No way to undo generically:** "Undo whatever the last button did" requires the remote to know how to reverse every kind of action — type-by-type — which it shouldn't.
3. **The invoker is coupled to the receivers:** `RemoteControl` imports `Light` and `Thermostat` directly. New devices mean editing the remote.

## The Solution: The Command Pattern

The Command pattern wraps a request — *who* should do *what*, with *which arguments* — into a standalone object that exposes `execute()` (and optionally `undo()`). The object that triggers the action holds a list of these command objects. The action becomes data: queueable, undoable, loggable, replayable.

The key participants are:

1. **Command (Interface):** Declares `execute()` and (often) `undo()`. Sometimes also `redo()` or `serialize()`.
2. **Concrete Commands:** Capture a *receiver*, the *parameters*, and any state needed to undo. Each one is a small object that knows how to do its specific job.
3. **Receiver:** The object that actually performs the work (the `Light`, the `Thermostat`, the `TextEditor`). The command's `execute()` calls the appropriate method on it.
4. **Invoker:** Holds and triggers commands. Doesn't know what they do — only that they have `execute()` / `undo()`. Maintains the history needed for undo/redo.
5. **Client:** Constructs the receivers and commands, wires them to invokers.

> **Why does this earn its weight?** Because once an action is an object, you get four capabilities for free: **undo** (call `undo()`), **macro** (a list of commands executed in order is itself a command), **queueing** (put commands on a thread-safe queue and a worker drains them), and **persistence** (serialize the command list to disk and replay later).

Two real-world scenarios in Python.

### Example 1: Smart-Home Remote with Undo

The remote holds a history stack. Each command captures its receiver and parameters; some commands also snapshot the previous state so they can undo correctly.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class Command(ABC):
    @abstractmethod
    def execute(self) -> None: ...
    @abstractmethod
    def undo(self) -> None: ...


class Light:
    def on(self)  -> None: print("Light ON")
    def off(self) -> None: print("Light OFF")


class Thermostat:
    def __init__(self) -> None:
        self.temperature = 20

    def set_temperature(self, temp: int) -> None:
        print(f"Thermostat set to {temp}°C")
        self.temperature = temp


class LightOnCommand(Command):
    def __init__(self, light: Light) -> None:
        self._light = light

    def execute(self) -> None: self._light.on()
    def undo(self)    -> None: self._light.off()


class LightOffCommand(Command):
    def __init__(self, light: Light) -> None:
        self._light = light

    def execute(self) -> None: self._light.off()
    def undo(self)    -> None: self._light.on()


class SetTemperatureCommand(Command):
    def __init__(self, thermostat: Thermostat, target: int) -> None:
        self._thermostat = thermostat
        self._target = target
        self._previous: int | None = None

    def execute(self) -> None:
        self._previous = self._thermostat.temperature
        self._thermostat.set_temperature(self._target)

    def undo(self) -> None:
        if self._previous is not None:
            self._thermostat.set_temperature(self._previous)


class RemoteControl:
    def __init__(self) -> None:
        self._history: list[Command] = []

    def execute(self, command: Command) -> None:
        command.execute()
        self._history.append(command)

    def undo_last(self) -> None:
        if self._history:
            self._history.pop().undo()
        else:
            print("Nothing to undo.")


if __name__ == "__main__":
    light, thermostat = Light(), Thermostat()
    remote = RemoteControl()

    print("--- Execute ---")
    remote.execute(LightOnCommand(light))
    remote.execute(SetTemperatureCommand(thermostat, 25))
    remote.execute(LightOffCommand(light))

    print("\n--- Undo ---")
    remote.undo_last()   # light back ON
    remote.undo_last()   # thermostat back to 20
    remote.undo_last()   # light back OFF
    remote.undo_last()   # nothing to undo
```

Note `SetTemperatureCommand`: a correct undo needs the *previous* temperature, not just the target. The command captures it during `execute()` and restores it during `undo()`. The remote doesn't need to know any of this — it just calls `undo()`.

### Example 2: Text Editor with Undo and Redo

A more demanding case. The invoker maintains *two* stacks — undo and redo. Every new command clears the redo stack, matching how editors actually work: redo only makes sense on the most recent undo path.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class TextEditor:
    def __init__(self) -> None:
        self.content = ""

    def append(self, text: str) -> None:
        self.content += text

    def delete_last(self, count: int) -> str:
        start = max(0, len(self.content) - count)
        deleted = self.content[start:]
        self.content = self.content[:start]
        return deleted


class EditorCommand(ABC):
    @abstractmethod
    def execute(self) -> None: ...
    @abstractmethod
    def undo(self) -> None: ...


class TypeCommand(EditorCommand):
    def __init__(self, editor: TextEditor, text: str) -> None:
        self._editor = editor
        self._text = text

    def execute(self) -> None:
        self._editor.append(self._text)
        print(f'Typed: "{self._text}"')

    def undo(self) -> None:
        self._editor.delete_last(len(self._text))
        print(f'Undo type: "{self._text}"')


class DeleteCommand(EditorCommand):
    def __init__(self, editor: TextEditor, count: int) -> None:
        self._editor = editor
        self._count = count
        self._deleted = ""

    def execute(self) -> None:
        self._deleted = self._editor.delete_last(self._count)
        print(f'Deleted: "{self._deleted}"')

    def undo(self) -> None:
        self._editor.append(self._deleted)
        print(f'Undo delete: restored "{self._deleted}"')


class EditorInvoker:
    def __init__(self) -> None:
        self._undo_stack: list[EditorCommand] = []
        self._redo_stack: list[EditorCommand] = []

    def execute(self, command: EditorCommand) -> None:
        command.execute()
        self._undo_stack.append(command)
        self._redo_stack.clear()       # new action invalidates the redo path

    def undo(self) -> None:
        if not self._undo_stack:
            print("Nothing to undo.")
            return
        command = self._undo_stack.pop()
        command.undo()
        self._redo_stack.append(command)

    def redo(self) -> None:
        if not self._redo_stack:
            print("Nothing to redo.")
            return
        command = self._redo_stack.pop()
        command.execute()
        self._undo_stack.append(command)


if __name__ == "__main__":
    editor = TextEditor()
    invoker = EditorInvoker()

    invoker.execute(TypeCommand(editor, "Hello"))
    invoker.execute(TypeCommand(editor, " World"))
    invoker.execute(TypeCommand(editor, "!"))
    print(f"Content: {editor.content!r}")

    print("\n--- Undo twice ---")
    invoker.undo()
    invoker.undo()
    print(f"Content: {editor.content!r}")

    print("\n--- Redo once ---")
    invoker.redo()
    print(f"Content: {editor.content!r}")

    print("\n--- New action clears redo ---")
    invoker.execute(DeleteCommand(editor, 3))
    print(f"Content: {editor.content!r}")
    invoker.redo()    # redo stack was cleared
```

`DeleteCommand` snapshots the deleted text inside `execute()` so `undo()` can restore it character-for-character — the *invoker* never sees the text. The undo logic remains generic because every command knows how to reverse itself.

## When to Use the Command Pattern

| Situation | Without Command | With Command |
|---|---|---|
| Need undo / redo | Action is hardcoded — no way to reverse generically | Each command knows how to undo itself |
| Need to queue or schedule actions | Actions are method calls — can't be data | Commands are objects — go on a queue |
| Same action triggered from many places (button, menu, shortcut) | Each trigger has its own copy | All triggers share one command object |
| Need to log or replay actions | No reified action to log | Commands serialize naturally |

## Conclusion

The Command pattern is the right tool when:

- **An action has a life beyond the moment it's triggered:** It needs to be undone, redone, queued, replayed, or audited.
- **The invoker shouldn't know the receiver:** A button doesn't care if it turns on a light or sends a message — only that the command has `execute()`.
- **The "action" is genuinely a unit:** It bundles receiver + arguments + (often) state for undo. That bundle is meaningful enough to give it a name.

The pattern's strength is **reification**. Once an action is an object, you can do all the things you do with objects: store it, copy it, send it across a network, write it to disk. The cost is one class per action — modest for the capabilities it unlocks.

A note on Python: a lambda or partial is a tempting shorthand for a command (`lambda: light.on()`). It works for *execute* but offers no place for *undo* without growing into something like a class anyway. When undo or persistence is on the table, the full pattern carries its weight; for a one-shot deferred call, a callable is enough.
