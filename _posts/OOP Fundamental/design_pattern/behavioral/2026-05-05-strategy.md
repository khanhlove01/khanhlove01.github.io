## The Problem

You have a class that *does something* — sort, format, calculate a discount, pick a route — and you discover that the right *how* depends on the situation. The naive approach is to add `if/elif` branches inside the class for every variant. That works for two cases, then becomes a swamp.

A concrete example: a text editor that needs to format text in different cases. The first attempt:

```python
class TextEditor:
    def publish_text(self, text: str, mode: str) -> None:
        if mode == "upper":
            print(text.upper())
        elif mode == "lower":
            print(text.lower())
        elif mode == "title":
            print(text.title())
        else:
            raise ValueError(f"Unknown mode: {mode!r}")
```

This compiles. It even works. The serious problems:

1. **The class knows about every variant:** Adding a new format means editing `TextEditor`. The class violates Open/Closed.
2. **Behaviors live as string flags:** `"upper"` / `"lower"` / `"title"` are stringly-typed and easy to mistype. The compiler/IDE can't help.
3. **Variants can't be configured or composed:** A variant that needs parameters (e.g., a member discount with a tier) has nowhere clean to put them — the `if` branch starts hauling extra arguments.

## The Solution: The Strategy Pattern

The Strategy pattern extracts each variant into its own class implementing a common interface. The original class — now the **Context** — holds a reference to a Strategy and delegates the variable behavior to it. Swapping strategies happens at runtime by setting a different object.

The key participants are:

1. **Strategy (Interface):** Declares the operation each variant must implement (e.g., `format(text)`, `sort(array)`).
2. **Concrete Strategies:** One class per variant. Each implements the operation its own way and can hold its own configuration.
3. **Context:** The class that uses a Strategy. It holds a Strategy reference and delegates to it. The Context exposes a setter so the strategy can be replaced.
4. **Client:** Picks a Strategy and gives it to the Context. The Context never knows which concrete strategy it has.

> **Strategy vs State:** Both delegate behavior to a separate object. With Strategy, the *caller* picks the variant: "use bubble sort." With State, the *object's own state* picks the variant: "I'm in the HasMoney state, so dispense." Strategy is configuration; State is a state machine.

Three real-world scenarios in Python.

### Example 1: Text Formatter

The minimal case. Three formatters share a common interface; the editor delegates to whichever one it's been given.

```python
from abc import ABC, abstractmethod


class TextFormatter(ABC):
    @abstractmethod
    def format(self, text: str) -> str: ...


class UpperCaseFormatter(TextFormatter):
    def format(self, text: str) -> str: return text.upper()


class LowerCaseFormatter(TextFormatter):
    def format(self, text: str) -> str: return text.lower()


class TitleCaseFormatter(TextFormatter):
    def format(self, text: str) -> str: return text.title()


class TextEditor:
    def __init__(self, formatter: TextFormatter) -> None:
        self._formatter = formatter

    def set_formatter(self, formatter: TextFormatter) -> None:
        self._formatter = formatter

    def publish_text(self, text: str) -> None:
        print(self._formatter.format(text))


if __name__ == "__main__":
    editor = TextEditor(UpperCaseFormatter())
    editor.publish_text("hello world")

    editor.set_formatter(TitleCaseFormatter())
    editor.publish_text("hello world")
```

Adding a new format — say, `KebabCaseFormatter` — is a single new class. `TextEditor` is never touched.

### Example 2: Sorting Algorithms

The canonical case. Several sort algorithms, each with different trade-offs (small arrays vs large, stable vs unstable). A `Sorter` context holds a strategy and the caller picks one based on the situation. A small helper auto-selects by data size.

```python
from abc import ABC, abstractmethod


class SortStrategy(ABC):
    @abstractmethod
    def sort(self, array: list[int]) -> list[int]: ...


class InsertionSortStrategy(SortStrategy):
    """O(n²) — best for small or nearly-sorted arrays."""
    def sort(self, array: list[int]) -> list[int]:
        arr = array.copy()
        for i in range(1, len(arr)):
            key, j = arr[i], i - 1
            while j >= 0 and arr[j] > key:
                arr[j + 1] = arr[j]
                j -= 1
            arr[j + 1] = key
        return arr


class MergeSortStrategy(SortStrategy):
    """O(n log n) stable — great for medium-large datasets."""
    def sort(self, array: list[int]) -> list[int]:
        return self._merge_sort(array.copy())

    def _merge_sort(self, arr: list[int]) -> list[int]:
        if len(arr) <= 1: return arr
        mid = len(arr) // 2
        return self._merge(self._merge_sort(arr[:mid]), self._merge_sort(arr[mid:]))

    def _merge(self, left: list[int], right: list[int]) -> list[int]:
        result, i, j = [], 0, 0
        while i < len(left) and j < len(right):
            if left[i] <= right[j]:
                result.append(left[i]); i += 1
            else:
                result.append(right[j]); j += 1
        result.extend(left[i:]); result.extend(right[j:])
        return result


class QuickSortStrategy(SortStrategy):
    """O(n log n) average — fastest in practice for large data."""
    def sort(self, array: list[int]) -> list[int]:
        arr = array.copy()
        self._quick_sort(arr, 0, len(arr) - 1)
        return arr

    def _quick_sort(self, arr: list[int], low: int, high: int) -> None:
        if low < high:
            p = self._partition(arr, low, high)
            self._quick_sort(arr, low, p - 1)
            self._quick_sort(arr, p + 1, high)

    def _partition(self, arr: list[int], low: int, high: int) -> int:
        pivot, i = arr[high], low - 1
        for j in range(low, high):
            if arr[j] <= pivot:
                i += 1
                arr[i], arr[j] = arr[j], arr[i]
        arr[i + 1], arr[high] = arr[high], arr[i + 1]
        return i + 1


class Sorter:
    def __init__(self, strategy: SortStrategy) -> None:
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy) -> None:
        self._strategy = strategy

    def sort_array(self, array: list[int]) -> list[int]:
        return self._strategy.sort(array)


def pick_strategy(array: list[int]) -> SortStrategy:
    n = len(array)
    if n <= 10:   return InsertionSortStrategy()
    if n <= 1000: return MergeSortStrategy()
    return QuickSortStrategy()


if __name__ == "__main__":
    sorter = Sorter(InsertionSortStrategy())
    print(sorter.sort_array([38, 27, 43, 3, 9, 82, 10]))

    big = list(range(2000, 0, -1))
    sorter.set_strategy(pick_strategy(big))
    print(sorter.sort_array(big)[:5], "...")
```

`pick_strategy` is the natural place for "choose the right algorithm for the data" logic — outside `Sorter`, easy to test in isolation, and easy to evolve.

### Example 3: E-commerce Discount Strategies

A real-world pricing engine where the discount rule depends on the situation: regular customer, member, bulk order, holiday sale. Each is a strategy with its own parameters; the cart delegates to whichever is active.

```python
from abc import ABC, abstractmethod


class DiscountStrategy(ABC):
    @abstractmethod
    def calculate_price(self, unit_price: float, quantity: int) -> float: ...

    @abstractmethod
    def describe(self) -> str: ...


class NoDiscount(DiscountStrategy):
    def calculate_price(self, unit_price: float, quantity: int) -> float:
        return unit_price * quantity

    def describe(self) -> str:
        return "Regular price"


class MemberDiscount(DiscountStrategy):
    TIERS = {"silver": 0.10, "gold": 0.20, "platinum": 0.30}

    def __init__(self, tier: str = "silver") -> None:
        self.tier = tier

    def calculate_price(self, unit_price: float, quantity: int) -> float:
        rate = self.TIERS.get(self.tier, 0.10)
        return unit_price * quantity * (1 - rate)

    def describe(self) -> str:
        rate = int(self.TIERS.get(self.tier, 0.10) * 100)
        return f"Member ({self.tier} tier — {rate}% off)"


class BulkDiscount(DiscountStrategy):
    def calculate_price(self, unit_price: float, quantity: int) -> float:
        if   quantity >= 20: rate = 0.25
        elif quantity >= 10: rate = 0.15
        elif quantity >= 5:  rate = 0.05
        else:                rate = 0.0
        return unit_price * quantity * (1 - rate)

    def describe(self) -> str:
        return "Bulk: 5%/15%/25% for 5+/10+/20+ units"


class SeasonalSale(DiscountStrategy):
    def __init__(self, name: str, rate: float) -> None:
        self.name, self.rate = name, rate

    def calculate_price(self, unit_price: float, quantity: int) -> float:
        return unit_price * quantity * (1 - self.rate)

    def describe(self) -> str:
        return f"{self.name} ({int(self.rate * 100)}% off)"


class ShoppingCart:
    def __init__(self, strategy: DiscountStrategy) -> None:
        self._strategy = strategy
        self._items: list[tuple[str, float, int]] = []

    def set_strategy(self, strategy: DiscountStrategy) -> None:
        self._strategy = strategy

    def add_item(self, name: str, unit_price: float, quantity: int) -> None:
        self._items.append((name, unit_price, quantity))

    def total(self) -> float:
        return sum(self._strategy.calculate_price(p, q) for _, p, q in self._items)


if __name__ == "__main__":
    cart = ShoppingCart(NoDiscount())
    cart.add_item("Headphones", 89.99, 1)
    cart.add_item("Cable",       9.99, 12)
    print(f"{cart._strategy.describe():<30} ${cart.total():.2f}")

    cart.set_strategy(MemberDiscount(tier="gold"))
    print(f"{cart._strategy.describe():<30} ${cart.total():.2f}")

    cart.set_strategy(BulkDiscount())
    print(f"{cart._strategy.describe():<30} ${cart.total():.2f}")

    cart.set_strategy(SeasonalSale("Holiday", 0.40))
    print(f"{cart._strategy.describe():<30} ${cart.total():.2f}")
```

Each strategy is its own object — `MemberDiscount` carries its tier, `SeasonalSale` carries its sale name and rate. Compare this to a single `if/elif` ladder where every parameter would have to flow as an argument: the Strategy version *encapsulates configuration with behavior*.

## When to Use the Strategy Pattern

| Situation | Without Strategy | With Strategy |
|---|---|---|
| Multiple ways to do the same operation | `if/elif` ladder in the context class | One class per variant; context delegates |
| Variants have configuration parameters | Extra arguments threaded through every branch | Each strategy holds its own config |
| Variant should be swappable at runtime | Mutable `mode` flag, fragile | `set_strategy(...)` — type-safe |
| Adding a new variant | Modify the context | Add one new class |

## Conclusion

The Strategy pattern is the right tool when:

- **The same operation has multiple implementations:** And the choice between them is a business or runtime decision, not a structural one.
- **You want to add variants without touching the existing class:** Each new variant is a new class implementing the interface. The context never changes.
- **A variant carries its own configuration:** Member tier, sale name, retry count — encapsulated with the behavior, not bolted on.

The pattern's strength is **isolation**. Every strategy is a self-contained object you can test, configure, and swap on its own. The context becomes thin — it picks an object and calls one method.

A note on Python: when a strategy has only one method and no state, a plain function (or `Callable`) is often enough. Reach for full Strategy classes when variants carry state, when you want a discoverable type hierarchy (e.g., for serialization), or when the operation is wide enough that a single function feels cramped.
