## The Problem

You have an object that behaves *differently* depending on its current state. The naive approach is to encode the state as a flag and write `if/elif` ladders inside every method. That works for two states; by four it's unmaintainable.

A concrete example: a vending machine. It has states like "idle", "item selected", "has money", "dispensing". The first attempt:

```python
class VendingMachine:
    def __init__(self) -> None:
        self.state = "idle"
        self.selected = ""

    def select_item(self, code: str) -> None:
        if self.state == "idle":
            self.selected = code
            self.state = "item_selected"
        elif self.state == "item_selected":
            self.selected = code
        elif self.state == "has_money":
            print("Cannot change item after inserting money.")
        elif self.state == "dispensing":
            print("Please wait.")

    def insert_coin(self, amount: float) -> None:
        if self.state == "idle":
            print("Select an item first.")
        elif self.state == "item_selected":
            self.state = "has_money"
        # ... etc, repeat for every state
```

This compiles. It even works at first. The serious problems:

1. **The ladder is repeated in every method:** `select_item`, `insert_coin`, `dispense_item` — each one branches on `self.state`. Adding a new state means editing every method.
2. **Transitions are scattered:** A transition like "ItemSelected → HasMoney" lives inside `insert_coin`. The full state graph is buried.
3. **Adding a state breaks everything:** Forget one branch and you get silent wrong behavior. There's no compile-time check that you've covered every state.

## The Solution: The State Pattern

The State pattern represents each state as **its own class** that implements a common state interface. The context (the object whose behavior is changing) holds a reference to a state object and *delegates every operation to it*. State transitions are performed by the state objects themselves: a state replaces itself on the context with the next state.

The key participants are:

1. **State (Interface):** Declares one method per operation the context exposes (e.g., `select_item`, `insert_coin`, `dispense_item`). Each method takes the context as an argument so the state can mutate it and trigger transitions.
2. **Concrete States:** One class per state. Each implements every operation according to what's allowed in *that* state, and triggers transitions by calling `context.set_state(NextState())`.
3. **Context:** Holds the current state and exposes a public API. Each public method is a one-liner that delegates to the current state. Also exposes `set_state` for the states to use.

> **State vs Strategy:** Both delegate behavior to a separate object. With Strategy, the *caller* picks the variant. With State, the *object's own state* picks it — and the object switches itself between states based on what happens. State is "the object behaves like a different object now"; Strategy is "I configured the object with this algorithm."

Three real-world scenarios in Python.

### Example 1: Vending Machine

The classic case. Four states, three operations. Each state spells out exactly what's allowed and what isn't, and triggers the next transition.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class MachineState(ABC):
    @abstractmethod
    def select_item(self, ctx: VendingMachine, code: str) -> None: ...

    @abstractmethod
    def insert_coin(self, ctx: VendingMachine, amount: float) -> None: ...

    @abstractmethod
    def dispense_item(self, ctx: VendingMachine) -> None: ...


class IdleState(MachineState):
    def select_item(self, ctx: VendingMachine, code: str) -> None:
        ctx.selected = code
        ctx.set_state(ItemSelectedState())

    def insert_coin(self, ctx: VendingMachine, amount: float) -> None:
        print("Select an item first.")

    def dispense_item(self, ctx: VendingMachine) -> None:
        print("No item selected.")


class ItemSelectedState(MachineState):
    def select_item(self, ctx: VendingMachine, code: str) -> None:
        ctx.selected = code  # change selection allowed

    def insert_coin(self, ctx: VendingMachine, amount: float) -> None:
        ctx.inserted = amount
        ctx.set_state(HasMoneyState())

    def dispense_item(self, ctx: VendingMachine) -> None:
        print("Insert a coin first.")


class HasMoneyState(MachineState):
    def select_item(self, ctx: VendingMachine, code: str) -> None:
        print("Cannot change item after inserting money.")

    def insert_coin(self, ctx: VendingMachine, amount: float) -> None:
        print("Money already inserted.")

    def dispense_item(self, ctx: VendingMachine) -> None:
        print(f"Dispensing {ctx.selected!r}.")
        ctx.reset()


class VendingMachine:
    def __init__(self) -> None:
        self._state: MachineState = IdleState()
        self.selected = ""
        self.inserted = 0.0

    def set_state(self, state: MachineState) -> None:
        self._state = state

    def reset(self) -> None:
        self.selected, self.inserted = "", 0.0
        self._state = IdleState()

    def select_item(self, code: str)    -> None: self._state.select_item(self, code)
    def insert_coin(self, amount: float) -> None: self._state.insert_coin(self, amount)
    def dispense_item(self)              -> None: self._state.dispense_item(self)


if __name__ == "__main__":
    vm = VendingMachine()
    vm.insert_coin(1.0)        # idle: rejected
    vm.select_item("A1")        # idle → item-selected
    vm.insert_coin(1.5)         # item-selected → has-money
    vm.select_item("B2")        # has-money: rejected
    vm.dispense_item()          # has-money → idle
```

Adding a new state — say, `OutOfStockState` — means writing one new class and pointing the relevant transitions at it. The existing states don't have to change.

### Example 2: Document Workflow

A document moves through *Draft → UnderReview → Published* and back. Each state has different rules about what edit/approve/reject/unpublish do. The state classes encode those rules directly.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class DocumentState(ABC):
    @abstractmethod
    def edit(self, doc: Document, content: str) -> None: ...
    @abstractmethod
    def submit_for_review(self, doc: Document) -> None: ...
    @abstractmethod
    def approve(self, doc: Document) -> None: ...
    @abstractmethod
    def reject(self, doc: Document) -> None: ...
    @abstractmethod
    def unpublish(self, doc: Document) -> None: ...


class DraftState(DocumentState):
    def edit(self, doc: Document, content: str) -> None:
        doc.content = content
    def submit_for_review(self, doc: Document) -> None:
        doc.set_state(UnderReviewState())
    def approve(self, doc: Document) -> None:
        print("Submit for review first.")
    def reject(self, doc: Document) -> None:
        print("Nothing to reject.")
    def unpublish(self, doc: Document) -> None:
        print("Already a draft.")


class UnderReviewState(DocumentState):
    def edit(self, doc: Document, content: str) -> None:
        print("Cannot edit while under review.")
    def submit_for_review(self, doc: Document) -> None:
        print("Already under review.")
    def approve(self, doc: Document) -> None:
        doc.set_state(PublishedState())
    def reject(self, doc: Document) -> None:
        doc.set_state(DraftState())
    def unpublish(self, doc: Document) -> None:
        print("Not published yet.")


class PublishedState(DocumentState):
    def edit(self, doc: Document, content: str) -> None:
        print("Unpublish before editing.")
    def submit_for_review(self, doc: Document) -> None:
        print("Already published.")
    def approve(self, doc: Document) -> None:
        print("Already published.")
    def reject(self, doc: Document) -> None:
        print("Cannot reject a published document.")
    def unpublish(self, doc: Document) -> None:
        doc.set_state(DraftState())


class Document:
    def __init__(self, title: str) -> None:
        self.title = title
        self.content = ""
        self._state: DocumentState = DraftState()

    def set_state(self, state: DocumentState) -> None:
        self._state = state

    def edit(self, content: str)     -> None: self._state.edit(self, content)
    def submit_for_review(self)      -> None: self._state.submit_for_review(self)
    def approve(self)                -> None: self._state.approve(self)
    def reject(self)                 -> None: self._state.reject(self)
    def unpublish(self)              -> None: self._state.unpublish(self)


if __name__ == "__main__":
    doc = Document("Annual Report")
    doc.edit("First draft.")
    doc.approve()             # blocked — still a draft
    doc.submit_for_review()
    doc.edit("Edits!")        # blocked — under review
    doc.reject()              # back to draft
    doc.edit("Revised.")
    doc.submit_for_review()
    doc.approve()             # published
    doc.unpublish()           # back to draft
```

The "what is allowed in this state" rule is local to each state class — easy to read, hard to violate. Permission gating is encoded by the state itself, not by scattered checks across the document class.

### Example 3: ATM with Multi-Step Authentication

A more demanding case: an ATM with PIN-attempt counting, lockout, and an out-of-service terminal state. The `CardInsertedState` carries its own private counter — *state objects can be stateful*.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class AtmState(ABC):
    @abstractmethod
    def insert_card(self, ctx: Atm) -> None: ...
    @abstractmethod
    def enter_pin(self, ctx: Atm, pin: str) -> None: ...
    @abstractmethod
    def withdraw(self, ctx: Atm, amount: float) -> None: ...
    @abstractmethod
    def report_error(self, ctx: Atm) -> None: ...


class IdleState(AtmState):
    def insert_card(self, ctx: Atm) -> None:
        ctx.set_state(CardInsertedState())
    def enter_pin(self, ctx: Atm, pin: str) -> None:
        print("No card inserted.")
    def withdraw(self, ctx: Atm, amount: float) -> None:
        print("No card inserted.")
    def report_error(self, ctx: Atm) -> None:
        ctx.set_state(OutOfServiceState())


class CardInsertedState(AtmState):
    def __init__(self) -> None:
        self._attempts = 0

    def insert_card(self, ctx: Atm) -> None:
        print("Card already inserted.")

    def enter_pin(self, ctx: Atm, pin: str) -> None:
        if pin == ctx.correct_pin:
            ctx.set_state(AuthenticatedState())
            return
        self._attempts += 1
        remaining = 3 - self._attempts
        print(f"Wrong PIN. {remaining} attempts left.")
        if self._attempts >= 3:
            print("Card ejected — too many wrong attempts.")
            ctx.set_state(IdleState())

    def withdraw(self, ctx: Atm, amount: float) -> None:
        print("Enter PIN first.")

    def report_error(self, ctx: Atm) -> None:
        ctx.set_state(OutOfServiceState())


class AuthenticatedState(AtmState):
    def insert_card(self, ctx: Atm) -> None:
        print("Card already inserted.")
    def enter_pin(self, ctx: Atm, pin: str) -> None:
        print("Already authenticated.")
    def withdraw(self, ctx: Atm, amount: float) -> None:
        if amount > ctx.balance:
            print(f"Insufficient funds (balance ${ctx.balance:.2f}).")
            return
        ctx.balance -= amount
        print(f"Dispensed ${amount:.2f}. Balance: ${ctx.balance:.2f}.")
    def report_error(self, ctx: Atm) -> None:
        ctx.set_state(OutOfServiceState())


class OutOfServiceState(AtmState):
    def insert_card(self, ctx: Atm)               -> None: print("Out of service.")
    def enter_pin(self, ctx: Atm, pin: str)        -> None: print("Out of service.")
    def withdraw(self, ctx: Atm, amount: float)    -> None: print("Out of service.")
    def report_error(self, ctx: Atm)               -> None: print("Already out of service.")


class Atm:
    def __init__(self, balance: float, pin: str = "1234") -> None:
        self.balance = balance
        self.correct_pin = pin
        self._state: AtmState = IdleState()

    def set_state(self, state: AtmState) -> None:
        self._state = state

    def insert_card(self)             -> None: self._state.insert_card(self)
    def enter_pin(self, pin: str)      -> None: self._state.enter_pin(self, pin)
    def withdraw(self, amount: float)  -> None: self._state.withdraw(self, amount)
    def report_error(self)             -> None: self._state.report_error(self)


if __name__ == "__main__":
    atm = Atm(balance=10_000.00, pin="1234")

    atm.insert_card()
    atm.enter_pin("bad1")
    atm.enter_pin("bad2")
    atm.enter_pin("bad3")   # ejected — back to idle

    atm.insert_card()
    atm.enter_pin("1234")
    atm.withdraw(3_000)
    atm.report_error()       # out of service
    atm.insert_card()        # rejected
```

The `_attempts` counter lives on `CardInsertedState`, not on the ATM. When the state transitions away (correct PIN or lockout), the counter is naturally discarded — no manual reset, no leaking state across sessions.

## When to Use the State Pattern

| Situation | Without State | With State |
|---|---|---|
| Behavior depends on a state field | `if/elif` ladder in every method | Each state encodes its own behavior |
| State transitions | Scattered across methods | Owned by state classes; one place per transition |
| Adding a new state | Edit every method | Add one new class |
| State-local data (e.g., attempt counters) | Lives on the context, must be reset manually | Lives on the state object, dies with the transition |

## Conclusion

The State pattern is the right tool when:

- **Behavior changes with internal state, not just configuration:** The same operation does different things depending on what state the object is in.
- **The state machine is non-trivial:** Three or more states, several operations, transitions that aren't obvious from one or two `if`s.
- **You want transitions in one place:** Each state class owns its outgoing transitions; the state graph is recoverable from the code without grep.

The pattern's strength is **closing the open-set of state**. Each state declares exactly what it allows and what it transitions to. Forgetting a case in a new state class is a missing-method error, not a silent fall-through.

A note: don't reach for State when a simple boolean and one `if` would do. The overhead is real — N+1 classes for N states. The pattern earns its weight when the matrix of `(state × operation)` becomes too big to keep straight in one method.
