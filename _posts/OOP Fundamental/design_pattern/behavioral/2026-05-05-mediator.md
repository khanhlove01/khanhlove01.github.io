## The Problem

You have a set of components that need to talk to each other. The naive approach is to give each one a reference to the others and let them call methods directly. With 3 components that's fine; at 6 you have 30 possible directed pairs and a tangle of mutual references.

A concrete example: a chat application. Each user needs to send to every other user. The first attempt:

```python
class ChatUser:
    def __init__(self, name: str) -> None:
        self.name = name
        self._peers: list[ChatUser] = []

    def add_peer(self, peer: ChatUser) -> None:
        self._peers.append(peer)

    def send(self, message: str) -> None:
        for peer in self._peers:
            peer.receive(message, self.name)

    def receive(self, message: str, sender: str) -> None:
        print(f"{self.name}: [{sender}] {message}")
```

This works for three users in a small room. Now add: muting, private messages, moderation rules, server-side logging. The serious problems:

1. **Quadratic wiring:** Every user must know every other user. Joining a 100-person room means 99 references — and removing a user means cleaning up references in 99 places.
2. **Logic lives in the wrong place:** "Block messages from muted users" has to be enforced *somewhere*. If it's in `send()`, every `ChatUser` carries the rule; if it's in `receive()`, every `ChatUser` re-checks. Neither is right.
3. **Variants spawn subclasses:** A moderated chat needs different rules; you start subclassing `ChatUser`, then you have `MutedChatUser`, `PrivateChatUser`, and the model collapses.

## The Solution: The Mediator Pattern

The Mediator pattern introduces a single hub — the **Mediator** — that knows about all the components, and each component talks only to the mediator. Components don't know each other; they tell the mediator "I want to do X" and the mediator decides what that means.

The key participants are:

1. **Mediator (Interface):** Declares the methods components use to communicate (e.g., `send_message(message, sender)`, `notify(device, event)`).
2. **Concrete Mediator:** Holds references to its components and implements the coordination logic. The rules, the routing, the validation — they all live here.
3. **Components (Colleagues):** Each one holds a reference to the mediator and only the mediator. To talk to other components, they go through it. They don't know who else exists.
4. **Client:** Constructs the mediator and components, registers components with it.

> **Mediator vs Observer:** Both decouple the source of an event from its consumers. Observer just *notifies*: the subject says "this happened" and observers react independently. Mediator *coordinates*: it interprets the event, applies rules, and sometimes routes it to a *specific* component (not all of them). Mediator centralizes *logic*; Observer centralizes *notification*.

Three real-world scenarios in Python.

### Example 1: Chat Room

A chat mediator that broadcasts messages to all members except the sender. The users only know the mediator interface — they don't hold references to each other. Variants of the mediator (private DM, moderated) implement different routing rules without changing the user classes.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class ChatMediator(ABC):
    @abstractmethod
    def add_user(self, user: User) -> None: ...
    @abstractmethod
    def send_message(self, message: str, sender: User) -> None: ...


class User:
    def __init__(self, name: str, mediator: ChatMediator) -> None:
        self.name = name
        self._mediator = mediator

    def send(self, message: str) -> None:
        self._mediator.send_message(message, self)

    def receive(self, message: str, sender: str) -> None:
        print(f"  {self.name}: [{sender}] {message}")


class ChatRoom(ChatMediator):
    """Broadcasts every message to all members except the sender."""
    def __init__(self, name: str) -> None:
        self.name = name
        self._users: list[User] = []

    def add_user(self, user: User) -> None:
        self._users.append(user)
        print(f"[{self.name}] {user.name} joined.")

    def send_message(self, message: str, sender: User) -> None:
        for user in self._users:
            if user is not sender:
                user.receive(message, sender.name)


class ModeratedChatRoom(ChatMediator):
    """Same broadcast, plus a mute list — muted senders are silently blocked."""
    def __init__(self, name: str) -> None:
        self.name = name
        self._users: list[User] = []
        self._muted: set[str] = set()

    def add_user(self, user: User) -> None:
        self._users.append(user)

    def mute(self, name: str) -> None:
        self._muted.add(name)
        print(f"[{self.name}] {name} muted.")

    def send_message(self, message: str, sender: User) -> None:
        if sender.name in self._muted:
            print(f"[{self.name}] message from {sender.name} blocked.")
            return
        for user in self._users:
            if user is not sender:
                user.receive(message, sender.name)


if __name__ == "__main__":
    room = ChatRoom("general")
    alice, bob, charlie = (User(n, room) for n in ("Alice", "Bob", "Charlie"))
    for u in (alice, bob, charlie):
        room.add_user(u)
    alice.send("Hi everyone!")

    print()
    mod_room = ModeratedChatRoom("moderated")
    troll = User("Troll99", mod_room)
    dave = User("Dave", mod_room)
    mod_room.add_user(troll); mod_room.add_user(dave)
    troll.send("first message — fine")
    mod_room.mute("Troll99")
    troll.send("blocked message")
```

Switching from a broadcast room to a moderated room means handing users a different mediator. The `User` class doesn't know — and doesn't care — which mediator it has.

### Example 2: Smart-Home Hub (Event-Driven Coordination)

A smart home where devices fire events and the hub decides what to do about them. The hub is the only place that knows the comfort rules ("hot temperature → AC on; motion detected → lights on; no motion → night mode"). Swapping the hub for an `EcoSmartHub` changes the rules without touching any device.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class SmartHomeMediator(ABC):
    @abstractmethod
    def register_device(self, device: SmartDevice) -> None: ...
    @abstractmethod
    def notify(self, device: SmartDevice, event: str) -> None: ...


class SmartDevice(ABC):
    def __init__(self, name: str, mediator: SmartHomeMediator) -> None:
        self.name = name
        self._mediator = mediator

    @abstractmethod
    def receive_command(self, command: str) -> None: ...


class Thermostat(SmartDevice):
    def __init__(self, name: str, mediator: SmartHomeMediator) -> None:
        super().__init__(name, mediator)
        self.temperature = 72

    def set_temperature(self, temp: int) -> None:
        self.temperature = temp
        print(f"  {self.name}: now {temp}°F")
        self._mediator.notify(self, "TEMP_HIGH" if temp > 75 else "TEMP_NORMAL")

    def receive_command(self, command: str) -> None:
        if command.startswith("SET:"):
            self.set_temperature(int(command.split(":")[1]))


class Lights(SmartDevice):
    def receive_command(self, command: str) -> None:
        print(f"  {self.name}: {command}")


class AirConditioner(SmartDevice):
    def receive_command(self, command: str) -> None:
        print(f"  {self.name}: {command}")


class MotionSensor(SmartDevice):
    def detect_motion(self) -> None:
        print(f"  {self.name}: motion detected")
        self._mediator.notify(self, "MOTION_DETECTED")

    def receive_command(self, command: str) -> None: ...


class SmartHomeHub(SmartHomeMediator):
    def __init__(self) -> None:
        self._devices: list[SmartDevice] = []

    def register_device(self, device: SmartDevice) -> None:
        self._devices.append(device)

    def _find(self, kind: type) -> SmartDevice | None:
        return next((d for d in self._devices if isinstance(d, kind)), None)

    def notify(self, device: SmartDevice, event: str) -> None:
        print(f"  [Hub] {event} from {device.name}")
        if event == "TEMP_HIGH":
            ac = self._find(AirConditioner)
            if ac: ac.receive_command("ON")
        elif event == "TEMP_NORMAL":
            ac = self._find(AirConditioner)
            if ac: ac.receive_command("OFF")
        elif event == "MOTION_DETECTED":
            lights = self._find(Lights)
            if lights: lights.receive_command("ON")


if __name__ == "__main__":
    hub = SmartHomeHub()
    thermostat = Thermostat("Living-Room Thermostat", hub)
    ac = AirConditioner("Central AC", hub)
    lights = Lights("Living-Room Lights", hub)
    motion = MotionSensor("Hallway Sensor", hub)

    for d in (thermostat, ac, lights, motion):
        hub.register_device(d)

    thermostat.set_temperature(80)   # TEMP_HIGH → AC ON
    thermostat.set_temperature(70)   # TEMP_NORMAL → AC OFF
    motion.detect_motion()           # MOTION_DETECTED → Lights ON
```

The thermostat has no idea the AC exists. The motion sensor has no idea about lights. Each device just *announces what happened*; the hub decides what that means.

### Example 3: Auction (Coordination with Validation)

A more demanding case: the mediator validates each bid against business rules — the new bid must beat the current highest by a minimum increment, and the bidder must have the funds. Bidders never see each other; the auctioneer sees all of them and enforces the rules.

```python
from __future__ import annotations
from abc import ABC, abstractmethod


class AuctionMediator(ABC):
    @abstractmethod
    def register_bidder(self, bidder: Bidder) -> None: ...
    @abstractmethod
    def place_bid(self, bidder: Bidder, amount: float) -> None: ...
    @abstractmethod
    def close_auction(self) -> None: ...


class Bidder:
    def __init__(self, name: str, balance: float, mediator: AuctionMediator) -> None:
        self.name = name
        self.balance = balance
        self._mediator = mediator

    def bid(self, amount: float) -> None:
        print(f"  {self.name} bids ${amount:.2f}")
        self._mediator.place_bid(self, amount)

    def receive(self, message: str) -> None:
        print(f"  -> {self.name}: {message}")


class Auctioneer(AuctionMediator):
    MINIMUM_INCREMENT = 10.0

    def __init__(self, item: str) -> None:
        self._item = item
        self._bidders: list[Bidder] = []
        self._highest = 0.0
        self._winner: Bidder | None = None

    def register_bidder(self, bidder: Bidder) -> None:
        self._bidders.append(bidder)

    def place_bid(self, bidder: Bidder, amount: float) -> None:
        min_required = self._highest + self.MINIMUM_INCREMENT
        if amount < min_required:
            bidder.receive(f"rejected — minimum is ${min_required:.2f}")
            return
        if amount > bidder.balance:
            bidder.receive(f"rejected — exceeds balance ${bidder.balance:.2f}")
            return

        self._highest = amount
        self._winner = bidder
        for b in self._bidders:
            if b is not bidder:
                b.receive(f"new highest bid ${amount:.2f} by {bidder.name}")

    def close_auction(self) -> None:
        if self._winner is None:
            for b in self._bidders:
                b.receive(f"closed — {self._item!r} unsold")
        else:
            for b in self._bidders:
                b.receive(f"closed — winner: {self._winner.name} at ${self._highest:.2f}")


if __name__ == "__main__":
    auction = Auctioneer("Vintage Watch")
    alice   = Bidder("Alice",   500.0, auction)
    bob     = Bidder("Bob",     300.0, auction)
    charlie = Bidder("Charlie", 700.0, auction)
    for b in (alice, bob, charlie):
        auction.register_bidder(b)

    alice.bid(100)        # accepted
    bob.bid(105)          # rejected — below increment
    bob.bid(150)          # accepted
    charlie.bid(200)      # accepted
    bob.bid(400)          # rejected — over balance
    auction.close_auction()
```

Validation lives in one place. Bidders never compare bids to each other; they just place a bid and learn whether it was accepted. Adding a `ReserveAuctioneer` (with a hidden reserve price) is a new mediator class — bidders don't change at all.

## When to Use the Mediator Pattern

| Situation | Without Mediator | With Mediator |
|---|---|---|
| N components talk to each other | N×N references | N references (each → mediator) |
| Coordination rules | Spread across components | Centralized in the mediator |
| Add a component | Wire it to every existing component | Register it with the mediator |
| Change the rules | Edit every component | Swap or edit the mediator |

## Conclusion

The Mediator pattern is the right tool when:

- **Several components need to coordinate:** And the coordination has *rules* — not just notification, but routing, validation, sequencing.
- **You don't want N×N coupling:** Components shouldn't import each other; they should import the mediator.
- **The rules might change:** A different mediator implementation gives the system a different personality without touching the components.

The pattern's strength is **centralization of coordination logic**. Each component stays small and focused; the mediator becomes the place where the application's *policy* lives. That trade-off is what makes the pattern valuable — and what makes it dangerous. A mediator is allowed to grow; a god-object is not. The line is whether the mediator's job stays "coordinate these components" or expands into "do everything."

A note on the relationship to Observer: many real systems use both. Components fire events (Observer) and a mediator subscribes to them, applying rules and routing as needed. That hybrid keeps notification cheap (no rules in the components) and rules in one place (the mediator).
