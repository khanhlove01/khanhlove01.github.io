## The Problem

You are building a system and you have a component that works perfectly — but its interface doesn't match what the rest of your code expects. You can't change the existing component (it's a third-party library, a legacy service, or code owned by another team). You also can't change your system's interface, because other parts depend on it.

A concrete example: your application expects all sensors to implement a `get_temperature()` method that returns Celsius, but a vendor's `FahrenheitSensor` only exposes `read_fahrenheit()`:

```python
class Thermometer(ABC):
    @abstractmethod
    def get_temperature(self) -> float:
        """Returns temperature in Celsius."""
        pass

class FahrenheitSensor:           # Third-party — cannot be changed
    def read_fahrenheit(self) -> float:
        return 98.6
```

If you try to use `FahrenheitSensor` directly where a `Thermometer` is expected, Python will raise an error at runtime — the interface simply doesn't match. The tempting workaround is to scatter conversion logic (subtract 32, multiply by 5/9) throughout the codebase every time you use the sensor. This has serious problems:

1. **Duplication:** The conversion formula is repeated in every call site. Change the formula and you must hunt down every usage.
2. **Tight coupling:** Every caller now knows about `read_fahrenheit()` — an implementation detail of the vendor class.
3. **Broken abstraction:** You lose the ability to swap sensors transparently, because callers can no longer treat all sensors uniformly.

## The Solution: The Adapter Pattern

The Adapter pattern introduces a **wrapper class** that sits between the client and the incompatible component. The adapter implements the interface the client expects, and internally translates each call into the equivalent call on the wrapped object.

The key participants are:

1. **Target (Interface):** The interface the client code expects (e.g., `Thermometer` with `get_temperature()`).
2. **Adaptee:** The existing class with an incompatible interface (e.g., `FahrenheitSensor` with `read_fahrenheit()`).
3. **Adapter:** Implements the Target interface and wraps an Adaptee instance. It translates each Target method call into the appropriate Adaptee call.
4. **Client:** Works exclusively with the Target interface — it never knows the Adaptee exists.

> **Object Adapter vs Class Adapter:** Python supports both. An *Object Adapter* (shown here) wraps an instance via composition — it holds a reference to the adaptee as a field. A *Class Adapter* uses multiple inheritance to inherit from both the target and the adaptee. The Object Adapter is far more common and preferred: it works at runtime, doesn't require the adaptee to be a class (duck typing), and doesn't suffer from diamond-inheritance issues.

Here are three real-world scenarios demonstrating the Adapter pattern in Python.

### Example 1: Temperature Sensor

A `Thermometer` abstract class defines the `get_temperature()` interface. A third-party `FahrenheitSensor` only speaks Fahrenheit. A `FahrenheitSensorAdapter` wraps it and translates the units — the client treats both sensors identically.

```python
from abc import ABC, abstractmethod


class Thermometer(ABC):
    @abstractmethod
    def get_temperature(self) -> float:
        """Returns temperature in Celsius."""
        pass


class CelsiusSensor(Thermometer):
    def get_temperature(self) -> float:
        return 25.0


class FahrenheitSensor:
    """Third-party class — incompatible interface, cannot be changed."""
    def read_fahrenheit(self) -> float:
        return 98.6


class FahrenheitSensorAdapter(Thermometer):
    def __init__(self, sensor: FahrenheitSensor) -> None:
        self._sensor = sensor

    def get_temperature(self) -> float:
        fahrenheit = self._sensor.read_fahrenheit()
        return (fahrenheit - 32) * 5.0 / 9.0


if __name__ == "__main__":
    sensors: list[Thermometer] = [
        CelsiusSensor(),
        FahrenheitSensorAdapter(FahrenheitSensor()),
    ]

    for sensor in sensors:
        print(f"Temperature: {sensor.get_temperature():.1f} C")
```

The client loop iterates over a `list[Thermometer]` — it never mentions `FahrenheitSensor` or `read_fahrenheit`. The conversion lives entirely inside the adapter, in one place.

### Example 2: Multi-Platform Notification Sender

Your application has a `NotificationSender` interface with a single `send(recipient, message)` method. You need to support Slack, Microsoft Teams, and Discord — each with a completely different SDK API. Adapters bridge every third-party client to the unified interface.

```python
from abc import ABC, abstractmethod


class NotificationSender(ABC):
    @abstractmethod
    def send(self, recipient: str, message: str) -> None:
        pass


# --- Third-party SDKs (cannot be changed) ---

class SlackClient:
    def post_message(self, channel: str, text: str, as_bot: bool) -> None:
        print(f"Slack -> #{channel}: {text} (bot={as_bot})")


class TeamsWebhook:
    def send_card(self, title: str, body: str, webhook_url: str) -> None:
        print(f"Teams -> {webhook_url}: [{title}] {body}")


class DiscordBot:
    def send_message(self, channel_id: int, content: str, tts: bool) -> None:
        print(f"Discord -> channel {channel_id}: {content} (tts={tts})")


# --- Adapters ---

class SlackAdapter(NotificationSender):
    def __init__(self, slack_client: SlackClient) -> None:
        self.slack_client = slack_client

    def send(self, recipient: str, message: str) -> None:
        self.slack_client.post_message(
            channel=recipient,
            text=message,
            as_bot=True,
        )


class TeamsAdapter(NotificationSender):
    def __init__(self, teams_webhook: TeamsWebhook, webhook_url: str) -> None:
        self.teams_webhook = teams_webhook
        self.webhook_url   = webhook_url

    def send(self, recipient: str, message: str) -> None:
        self.teams_webhook.send_card(
            title=recipient,
            body=message,
            webhook_url=self.webhook_url,
        )


class DiscordAdapter(NotificationSender):
    def __init__(self, discord_bot: DiscordBot, channel_id: int) -> None:
        self.discord_bot = discord_bot
        self.channel_id  = channel_id

    def send(self, recipient: str, message: str) -> None:
        self.discord_bot.send_message(
            channel_id=self.channel_id,
            content=message,
            tts=False,
        )


if __name__ == "__main__":
    senders: list[NotificationSender] = [
        SlackAdapter(SlackClient()),
        TeamsAdapter(TeamsWebhook(), "https://webhook.teams.com/abc"),
        DiscordAdapter(DiscordBot(), 123456789),
    ]

    senders[0].send("general",    "Build succeeded")
    senders[1].send("Deployment", "Service deployed to production")
    senders[2].send("alerts",     "CPU usage above 90%")
```

Adding a new platform — say, PagerDuty — requires only one new adapter class. The client loop and the `NotificationSender` interface remain completely unchanged.

### Example 3: JSON ↔ XML Service Bridge

Two services exist in the system: `JsonService` that speaks JSON, and `XmlService` that speaks XML. Each has clients expecting the other's format. Two bidirectional adapters — `JsonToXmlAdapter` and `XmlToJsonAdapter` — bridge the gap, with lightweight conversion helpers handling the format translation.

```python
import re
from abc import ABC, abstractmethod


class JsonApi(ABC):
    @abstractmethod
    def send_json(self, json_str: str) -> None:
        pass

    @abstractmethod
    def receive_json(self) -> str:
        pass


class XmlApi(ABC):
    @abstractmethod
    def send_xml(self, xml_str: str) -> None:
        pass

    @abstractmethod
    def receive_xml(self) -> str:
        pass


class JsonService(JsonApi):
    def send_json(self, json_str: str) -> None:
        print(f"JsonService: Sent {json_str}")

    def receive_json(self) -> str:
        return '{"status":"ok"}'


class XmlService(XmlApi):
    def send_xml(self, xml_str: str) -> None:
        print(f"XmlService: Sent {xml_str}")

    def receive_xml(self) -> str:
        return "<status>ok</status>"


# --- Conversion helpers ---

def _xml_to_json(xml_str: str) -> str:
    """Convert a single-element XML string like <key>value</key> to {"key":"value"}."""
    match = re.fullmatch(r"<(\w+)>(.*?)</\1>", xml_str.strip())
    if not match:
        raise ValueError(f"Cannot parse XML: {xml_str!r}")
    key, value = match.group(1), match.group(2)
    return f'{{"{key}":"{value}"}}'


def _json_to_xml(json_str: str) -> str:
    """Convert a single-key JSON string like {"key":"value"} to <key>value</key>."""
    match = re.search(r'"(\w+)"\s*:\s*"([^"]*)"', json_str.strip())
    if not match:
        raise ValueError(f"Cannot parse JSON: {json_str!r}")
    key, value = match.group(1), match.group(2)
    return f"<{key}>{value}</{key}>"


# --- Adapters ---

class JsonToXmlAdapter(XmlApi):
    """Makes a JsonApi look like an XmlApi — converts every XML call into JSON."""

    def __init__(self, json_api: JsonApi) -> None:
        self.json_api = json_api

    def send_xml(self, xml_str: str) -> None:
        self.json_api.send_json(_xml_to_json(xml_str))

    def receive_xml(self) -> str:
        return _json_to_xml(self.json_api.receive_json())


class XmlToJsonAdapter(JsonApi):
    """Makes an XmlApi look like a JsonApi — converts every JSON call into XML."""

    def __init__(self, xml_api: XmlApi) -> None:
        self.xml_api = xml_api

    def send_json(self, json_str: str) -> None:
        self.xml_api.send_xml(_json_to_xml(json_str))

    def receive_json(self) -> str:
        return _xml_to_json(self.xml_api.receive_xml())


if __name__ == "__main__":
    json_service = JsonService()
    xml_service  = XmlService()

    print("--- Use JSON service where XML is expected (JsonToXmlAdapter) ---")
    json_as_xml = JsonToXmlAdapter(json_service)
    json_as_xml.send_xml("<data>hello</data>")
    print(f"Received: {json_as_xml.receive_xml()}")

    print("\n--- Use XML service where JSON is expected (XmlToJsonAdapter) ---")
    xml_as_json = XmlToJsonAdapter(xml_service)
    xml_as_json.send_json('{"data":"hello"}')
    print(f"Received: {xml_as_json.receive_json()}")
```
{% endraw %}

This example shows that adapters work in both directions. `JsonToXmlAdapter` lets an XML client use a JSON service; `XmlToJsonAdapter` lets a JSON client use an XML service. Neither service class is touched.

## When to Use the Adapter Pattern

| Situation | Without Adapter | With Adapter |
|---|---|---|
| Third-party library has wrong interface | Scatter conversion logic everywhere | One adapter class isolates all translation |
| Legacy code with old interface | Rewrite the legacy class (risky) | Wrap it in an adapter, keep it untouched |
| Multiple incompatible providers | `if/elif` chains in every caller | One adapter per provider, uniform interface |
| Adding a new provider | Modify every call site | Write one new adapter class |

## Conclusion

The Adapter pattern is the right tool when:

- **You can't change the adaptee:** It's a third-party library, a legacy class, or code from another team.
- **You can't change the client interface:** Other components depend on it and changing it would be a breaking change.
- **You have multiple incompatible providers:** Instead of branching logic scattered across callers, each provider gets its own adapter, and all callers use the uniform interface.

The pattern's strength is **isolation** — all translation logic lives in one small class. When a vendor changes their API, you update exactly one adapter. When you add a new provider, you write exactly one new adapter. The rest of the system never knows anything changed.
