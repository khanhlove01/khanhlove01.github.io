## The Problem

Your system has a subsystem made of many cooperating classes. Each class is fine on its own, but using them together requires the client to know **the right calls in the right order, with the right error handling**. The subsystem leaks its complexity outward into every caller.

A concrete example: a smart home with three independent subsystems — lights, thermostat, security:

```python
class SmartLightsSystem:
    def on(self):  ...
    def off(self): ...

class Thermostat:
    def set_mode(self, mode: str):     ...
    def set_temperature(self, t: int): ...

class SecuritySystem:
    def arm(self):    ...
    def disarm(self): ...
```

When the user leaves the house, the client must remember to:

1. Turn the lights off.
2. Set the thermostat to "eco" mode.
3. Drop the temperature to 18°C.
4. Arm the security system.

Now imagine that logic scattered across every place "leaving home" is triggered: a button in the app, a Siri shortcut, a scheduled job. This has serious problems:

1. **Coupling:** Every caller knows about every subsystem class. Changing a subsystem (e.g., the security API now requires a PIN) ripples through all callers.
2. **Order is invisible:** Nothing prevents a caller from arming security *before* turning off lights, leaving rooms lit. The required sequence lives only in someone's head.
3. **No single source of truth:** When the workflow changes, you must hunt down every place that performs it.

## The Solution: The Facade Pattern

The Facade pattern introduces a **single high-level class** that exposes a simple interface for common workflows. The facade holds references to the subsystem objects and orchestrates them internally. Clients call one method on the facade — the subsystem stays out of sight.

The key participants are:

1. **Subsystem Classes:** The existing components doing real work. They are unchanged by the facade — they don't even know it exists.
2. **Facade:** A class that exposes a small, task-oriented API (`leave_home()`, `place_order()`). Each method orchestrates one or more subsystem calls in the correct order, with the right parameters.
3. **Client:** Talks only to the facade. It doesn't import the subsystem classes.

> **Facade is not a wrapper around one class.** That's just an alias. A facade earns its name by hiding multiple subsystems behind a single, higher-level interface — the value is in the *coordination*, not the indirection.

Three real-world scenarios in Python.

### Example 1: Smart Home

A `SmartHomeFacade` exposes two task-oriented methods, `leave_home()` and `arrive_home()`. Each one orchestrates the lights, thermostat, and security in the correct order.

```python
class SmartLightsSystem:
    def on(self):  print("Lights: on.")
    def off(self): print("Lights: off.")


class Thermostat:
    def __init__(self) -> None:
        self._mode = ""

    def set_mode(self, mode: str) -> None:
        self._mode = mode

    def set_temperature(self, degrees: int) -> None:
        print(f"Thermostat: mode={self._mode}, temp={degrees}C.")


class SecuritySystem:
    def arm(self):    print("Security: armed.")
    def disarm(self): print("Security: disarmed.")


class SmartHomeFacade:
    def __init__(self, lights: SmartLightsSystem,
                 thermostat: Thermostat, security: SecuritySystem) -> None:
        self._lights     = lights
        self._thermostat = thermostat
        self._security   = security

    def leave_home(self) -> None:
        self._lights.off()
        self._thermostat.set_mode("eco")
        self._thermostat.set_temperature(18)
        self._security.arm()

    def arrive_home(self) -> None:
        self._lights.on()
        self._thermostat.set_mode("comfort")
        self._thermostat.set_temperature(22)
        self._security.disarm()


if __name__ == "__main__":
    home = SmartHomeFacade(SmartLightsSystem(), Thermostat(), SecuritySystem())
    home.leave_home()
    home.arrive_home()
```

Every caller — the app button, the Siri shortcut, the scheduled job — now calls `home.leave_home()`. The order, the temperature, the eco mode are decided in one place. If "leaving home" later means *also* lowering the blinds, you change the facade and every caller picks it up for free.

### Example 2: Travel Booking

Booking a trip means calling three separate services in sequence — flight, hotel, car. A `TravelBookingFacade` exposes `book_trip()` and `cancel_trip()` and tracks the IDs internally so cancellation works correctly.

```python
class FlightBooking:
    def book_flight(self, destination: str, date: str) -> str:
        print(f"Flight: booked to {destination} on {date}.")
        return "FL-1001"

    def cancel_flight(self, confirmation_id: str) -> None:
        print(f"Flight: cancelled {confirmation_id}.")


class HotelReservation:
    def reserve_room(self, destination: str, check_in: str, check_out: str) -> str:
        print(f"Hotel: reserved in {destination} {check_in} -> {check_out}.")
        return "HT-2001"

    def cancel_reservation(self, confirmation_id: str) -> None:
        print(f"Hotel: cancelled {confirmation_id}.")


class CarRental:
    def rent_car(self, location: str, pickup: str, ret: str) -> str:
        print(f"Car: rented in {location} {pickup} -> {ret}.")
        return "CR-3001"

    def cancel_rental(self, confirmation_id: str) -> None:
        print(f"Car: cancelled {confirmation_id}.")


class TravelBookingFacade:
    def __init__(self, flights: FlightBooking,
                 hotels: HotelReservation, cars: CarRental) -> None:
        self._flights, self._hotels, self._cars = flights, hotels, cars
        self._flight_id = self._hotel_id = self._car_id = ""

    def book_trip(self, destination: str, check_in: str, check_out: str) -> None:
        self._flight_id = self._flights.book_flight(destination, check_in)
        self._hotel_id  = self._hotels.reserve_room(destination, check_in, check_out)
        self._car_id    = self._cars.rent_car(destination, check_in, check_out)

    def cancel_trip(self) -> None:
        self._flights.cancel_flight(self._flight_id)
        self._hotels.cancel_reservation(self._hotel_id)
        self._cars.cancel_rental(self._car_id)


if __name__ == "__main__":
    facade = TravelBookingFacade(FlightBooking(), HotelReservation(), CarRental())
    facade.book_trip("Paris", "2026-06-01", "2026-06-07")
    facade.cancel_trip()
```

The facade also acts as a small piece of *state* — it remembers the confirmation IDs from the booking so the cancellation method doesn't need them as arguments. That's a natural place for it; the client shouldn't have to track three IDs to cancel one trip.

### Example 3: E-commerce Order with Rollback

A more demanding case: placing an order touches inventory, payment, shipping, and notifications. Each step can fail, and a failure has to roll back earlier steps to avoid a partial order. The facade owns the orchestration *and* the compensation logic.

```python
class InventoryService:
    def __init__(self) -> None:
        self._stock = {"SKU-001": 5}

    def check_stock(self, product_id: str, qty: int) -> bool:
        return self._stock.get(product_id, 0) >= qty

    def reserve_stock(self, product_id: str, qty: int) -> None:
        self._stock[product_id] -= qty
        print(f"Inventory: reserved {qty} of {product_id}.")

    def release_stock(self, product_id: str, qty: int) -> None:
        self._stock[product_id] += qty
        print(f"Inventory: released {qty} of {product_id}.")


class PaymentService:
    def charge(self, method: str, amount: float) -> str | None:
        if method == "expired-card":
            print(f"Payment: failed for {method}.")
            return None
        print(f"Payment: charged ${amount} to {method}.")
        return "TXN-4001"


class ShippingService:
    def create_shipment(self, product_id: str, address: str) -> str:
        print(f"Shipping: created for {product_id} -> {address}.")
        return "TRK-5001"


class NotificationService:
    def send_order_confirmation(self, email: str, order_id: str, tracking: str) -> None:
        print(f"Notification: order {order_id} ({tracking}) sent to {email}.")


class OrderFacade:
    def __init__(self, inventory: InventoryService, payment: PaymentService,
                 shipping: ShippingService, notification: NotificationService) -> None:
        self._inventory    = inventory
        self._payment      = payment
        self._shipping     = shipping
        self._notification = notification

    def place_order(self, product_id: str, qty: int, method: str,
                    amount: float, address: str, email: str) -> bool:
        if not self._inventory.check_stock(product_id, qty):
            print("Order failed: out of stock.")
            return False

        self._inventory.reserve_stock(product_id, qty)

        transaction_id = self._payment.charge(method, amount)
        if transaction_id is None:
            self._inventory.release_stock(product_id, qty)
            return False

        tracking = self._shipping.create_shipment(product_id, address)
        self._notification.send_order_confirmation(email, "ORD-6001", tracking)
        return True


if __name__ == "__main__":
    facade = OrderFacade(InventoryService(), PaymentService(),
                         ShippingService(), NotificationService())
    facade.place_order("SKU-001", 2, "credit-card", 49.99, "123 Main St", "u@x.com")
    facade.place_order("SKU-001", 1, "expired-card", 29.99, "456 Oak Ave", "u@x.com")
```

The rollback (`release_stock` after a failed payment) lives entirely inside the facade. Callers see one boolean: did the order succeed? They never have to think about which steps need undoing.

## When to Use the Facade Pattern

| Situation | Without Facade | With Facade |
|---|---|---|
| Multi-step workflow over several services | Each caller orchestrates the steps | One facade method per workflow |
| Order or rollback rules | Repeated everywhere, easy to break | Centralized in one place |
| Subsystem internals leak into callers | Clients import 5 subsystem classes | Clients import only the facade |
| Subsystem API changes | Update every caller | Update the facade |

## Conclusion

The Facade pattern is the right tool when:

- **A workflow spans several classes:** And every caller of the workflow has to know the same set of steps in the same order.
- **You want to decouple clients from subsystem internals:** Clients should depend on intent (`place_order`, `leave_home`), not on the cast of subsystem classes that implement it.
- **There's compensation or error-handling logic:** That logic belongs in one place — the facade — not duplicated across every caller.

The pattern's strength is **simplification through aggregation**. The subsystem keeps its full power; advanced clients can still bypass the facade and use the subsystem directly when they need to. But the *common* path becomes a single method call, and the orchestration knowledge lives in exactly one class.

A note: a facade is allowed to be opinionated. It is not meant to expose every subsystem feature — that would defeat the purpose. A good facade picks the 80% case and exposes it cleanly; everything else stays available to clients who need it.
