**SOLID** is a set of five design guidelines (from Robert C. Martin and others) that help you keep modules **focused**, **extensible**, **safe to substitute**, **narrow at boundaries**, and **decoupled from infrastructure**. They overlap with broader ideas in **[design principles](/posts/design-principle/)**—SOLID just names five recurring moves.

Below: **what each letter stands for**, **what to watch for**, and a **small contrast** (usually violate vs align). Material is aligned with a personal **`4.Solid`** study folder.

> Note: the first letter is **SRP** (**S**ingle **R**esponsibility), not “SPP.”

---

## 1. SRP — Single Responsibility Principle

**Definition:** A module (class/function/package) should have **one reason to change**—one coherent job in the product sense (e.g. “calculate pay” vs “send email”), not one line of code.

**Characteristics**

- If you describe the type with **“and”** too often, split candidates appear.
- **Testing** gets easier when dependencies (DB, SMTP) are not glued to domain data.
- “One responsibility” is **about change drivers**, not counting methods.

**Violates SRP — “god” employee**

```python
class EmployeeGod:
    def __init__(self, name: str, email: str, base: float):
        self.name, self.email, self.base = name, email, base

    def net_pay(self) -> float: ...
    def save_to_db(self) -> None: ...
    def payslip_html(self) -> str: ...
    def email_payslip(self) -> None: ...
```

Payroll rules, persistence, formatting, and delivery all **change for different reasons**.

**Aligns with SRP — split by reason to change**

```python
class Employee:
    def __init__(self, name: str, email: str, base: float):
        self.name, self.email, self.base = name, email, base


class PayrollCalculator:
    def net_pay(self, emp: Employee) -> float: ...


class EmployeeRepository:
    def save(self, emp: Employee) -> None: ...
```

## 2. OCP — Open/Closed Principle

**Definition:** Types should be **open for extension** (new behavior without editing old core logic) and **closed for modification** (stable code paths stay untouched).

**Characteristics**

- Achieved with **abstractions** (ABC, `Protocol`) and **polymorphism**: add a **new class**, not new `elif` chains.
- Pair with **[realization](/posts/class-relationship/)** in diagrams: new implementors, same interface.

**Violates OCP — growing `if/else`**

```python
def charge(kind: str, amount: float) -> None:
    if kind == "card":
        ...
    elif kind == "paypal":
        ...
    # every new payment edits this function
```

**Aligns with OCP — extend via types**

```python
class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount: float) -> None: ...


class CardPayment(PaymentMethod):
    def pay(self, amount: float) -> None: ...


class PayPalPayment(PaymentMethod):
    def pay(self, amount: float) -> None: ...


def checkout(method: PaymentMethod, amount: float) -> None:
    method.pay(amount)  # unchanged when you add CryptoPayment
```

## 3. LSP — Liskov Substitution Principle

**Definition:** If `S` is a subtype of `T`, **anywhere you use `T`**, you should be able to use **`S` without surprises**—no broken contracts, no “works unless it’s this subclass.”

**Characteristics**

- **Behavioral** subtyping: `is-a` in code is stricter than English “is-a.”
- Subclasses that **weaken guarantees** (e.g. `save` always worked; now it raises) break clients.
- Classic pitfall: **`Square` extends `Rectangle`** with coupled width/height rules.

**Violates LSP — read-only document breaks `save`**

```python
class Document:
    def save(self, data: str) -> None:
        self.data = data


class ReadOnlyDocument(Document):
    def save(self, data: str) -> None:
        raise RuntimeError("read only")  # callers of Document.save break
```

**Aligns with LSP — separate types or shared contract**

```python
class Readable(ABC):
    @abstractmethod
    def body(self) -> str: ...


class Editable(Readable, ABC):
    @abstractmethod
    def save(self, data: str) -> None: ...
```

Or keep one hierarchy but **don’t promise** `save` where it cannot hold.

## 4. ISP — Interface Segregation Principle

**Definition:** Clients should not be **forced to depend on methods they do not use**—prefer **small, role-based** interfaces over one fat “god interface.”

**Characteristics**

- Design from **callers’** needs; avoid `NotImplementedError` stubs.
- Smaller ABCs / protocols **compose** better than a single mega-class.

**Violates ISP — fat media player**

```python
class MediaPlayer(ABC):
    @abstractmethod
    def play_audio(self, path: str) -> None: ...
    @abstractmethod
    def play_video(self, path: str) -> None: ...
    @abstractmethod
    def subtitles(self, path: str) -> None: ...


class Radio(MediaPlayer):
    def play_video(self, path: str) -> None:
        raise NotImplementedError  # forced junk
```

**Aligns with ISP — split roles**

```python
class AudioOutput(ABC):
    @abstractmethod
    def play_audio(self, path: str) -> None: ...


class VideoOutput(ABC):
    @abstractmethod
    def play_video(self, path: str) -> None: ...


class Radio(AudioOutput):
    def play_audio(self, path: str) -> None: ...
```

## 5. DIP — Dependency Inversion Principle (and dependency injection)

**Definition:**

1. **High-level** policy (business rules) should **not depend on low-level** details (SMTP, SQL driver, REST SDK).
2. **Both** should depend on **abstractions** (interfaces / ABCs / protocols).
3. Abstractions belong to the **high-level story**; **concrete adapters** implement them.

**Why “inversion”?** Without DIP, dependency arrows point **down** (policy → Gmail). With DIP, both policy and Gmail **point at** a shared **`EmailSender`** abstraction—the **direction of control** stays high-level, but the **source code dependencies** aim at the stable contract.

**What does “inject” mean?**  
**Dependency injection** means the **caller or framework supplies** an implementation of that abstraction—usually via the **constructor** (**constructor injection**), or sometimes setters / factories. The service **does not `new` the concrete mailer** inside itself; it **receives** an `EmailSender`, so tests can pass a **fake** and production can pass **Gmail** or **SendGrid**.

**Characteristics**

- High-level class takes **`Protocol` / ABC`** in `__init__` or method params.
- **Composition root** (FastAPI `Depends`, your `main`, DI container) wires real implementations once.
- DIP is the **principle**; injection is a common **mechanism** to satisfy it.

### Example A — violates DIP (hard-coded Gmail)

```python
class GmailApi:
    def send_gmail(self, to: str, subject: str, body: str) -> None:
        print("gmail", to, subject)


class WelcomeService:
    def __init__(self) -> None:
        self.mail = GmailApi()  # policy depends on concrete detail

    def run(self, email: str, name: str) -> None:
        self.mail.send_gmail(email, f"Hi {name}", "welcome")
```

Switching to Outlook **edits** `WelcomeService`.

### Example B — follows DIP + constructor injection

```python
from abc import ABC, abstractmethod


class EmailClient(ABC):
    @abstractmethod
    def send(self, to: str, subject: str, body: str) -> None: ...


class GmailClient(EmailClient):
    def send(self, to: str, subject: str, body: str) -> None:
        print("[gmail]", to, subject)


class OutlookClient(EmailClient):
    def send(self, to: str, subject: str, body: str) -> None:
        print("[outlook]", to, subject)


class WelcomeService:
    def __init__(self, mail: EmailClient):
        self.mail = mail  # injected — depends on abstraction only

    def run(self, email: str, name: str) -> None:
        self.mail.send(email, f"Hi {name}", "welcome")


# wiring (composition root)
svc = WelcomeService(GmailClient())
svc.run("a@b.com", "Ann")

test_svc = WelcomeService(OutlookClient())  # or a MockEmailClient in tests
```

### Example C — repository DIP (same pattern, different detail)

```python
class UserRepository(ABC):
    @abstractmethod
    def by_id(self, user_id: str) -> dict | None: ...


class SqlUserRepository(UserRepository):
    def by_id(self, user_id: str) -> dict | None:
        return {"id": user_id, "name": "from SQL"}


class GetUserProfile:
    def __init__(self, users: UserRepository):
        self.users = users

    def handle(self, user_id: str) -> str:
        u = self.users.by_id(user_id)
        return u["name"] if u else "unknown"
```

`GetUserProfile` stays unchanged when you add `RedisUserRepository` or an in-memory fake.

## SOLID at a glance

| Letter | Principle | Core question | Typical lever |
|--------|-----------|---------------|---------------|
| **S** | Single Responsibility | What is the **one** reason this module changes? | Split by concern |
| **O** | Open/Closed | Can I **add** behavior without **editing** stable code? | Abstractions + new types |
| **L** | Liskov Substitution | Can I swap subclasses **without** breaking callers? | Honest contracts |
| **I** | Interface Segregation | Does this client see **only** methods it needs? | Smaller interfaces |
| **D** | Dependency Inversion | Does policy depend on **SMTP/SQL**, or on a **port**? | ABC/`Protocol` + **inject** adapters |
